Глава 1. ОБЩИЕ ПОЛОЖЕНИЯ
Structured Query Language – Язык структурированных запросов

1. Весь мир SQL – это таблицы с данными, где данные в разных таблицах явно или неявно связаны между собой.
Таблицы — это изолированные острова данных. SQL — это мост между островами. 
Виды связей между таблицами:
Явные связи – прописаны в схеме («живые индексы») – PK/FK (Primary Key / Foreign Key) – ID. (связь может быть «родственная» – одно рождает другое или дружеская – связь независимая»);
Неявные связи – формируются в результате логического сопоставления данных из столбцов (одного формата) между таблицами, и их последующей комбинированной связке.

2. В таблице, по природе, есть колонка и строка:
Колонка: один столбец = одна характеристика с одним форматом данных (жесткая дисциплина);
Строка: набор характеристик разных форматов, скрепленных в одну горизонталь. В идеальном сценарии имеет ID, но ID — это всего лишь гарант уникальности, нет ID - уникальность формируется комбинацией признаков (комбинацией колонок).

3. Основная роль SQL в аналитике заключается в приказах, мы приказываем системе показать, что мы хотим видеть, в каком виде (трансформация) предоставить данные, откуда взять эти данные, а также отфильтровать некоторые данные – отсев ненужного.
	Иначе говоря: 
SELECT – что мы хотим видеть (вывод данных, тут же - математика с данными, смена имен колонок, смена форматов данных, команды по «зачистке шума», функции агрегации, оконные функцию), в этом блоке мы создаем виртуальную проекцию (в оперативной памяти RAM), которая не меняет оригинал.
FROM – откуда взять (из какого хранилища);
WHERE – задать фильтрацию (задаем условия, где система режет сырой поток данных, отсеивая лишнее, до того, как они пройдут в активную фазу обработки данных). *активная фаза – данные попали в RAM.


4. Структура построения приказа имеет строгую последовательность:
	БЛОК_1: SELECT (что показать)
	БЛОК_2: FROM (адрес данных)
	БЛОК_3: JOIN ON (виртуальное соединение строк из таблиц: «создание одной сверх-таблицы»)
/JOIN синтаксически не может существовать без FROM. Он является его прямым расширением.
БЛОК_4: WHERE (фильтр данных, или «сито для сырых данных» до активной фазы чтения RAM)
БЛОК_5: GROUP BY (сжатие одноименных строк в одну (создание групп) для проведения расчетов внутри этих групп)
БЛОК_6: HAVING (фильтр-весы для групп) – существует ТОЛЬКО с group by
БЛОК_7: ORDER BY (сортировка) – финальные штрихи
БЛОК_8: LIMIT N (вывод ограниченного числа строк, заданных оператором в N)
/Закон: Пропуск любого блока допустим, но изменение их порядка — запрещено.

5. Система выполняет приказ в определённой последовательности, не равной структуре приказу:
Порядок исполнения запроса системой:
1.	FROM + JOIN ON (Выбор мира данных и его сборка воедино)
2.	WHERE (Первичный отсев до активной фазы выполнения в  RAM)
3.	GROUP BY (Схлопывание одноименных строк в группы для проведения расчетов внутри этих самых групп)
4.	HAVING (Отсев групп по результатам функций агрегаций) – существует только вместе с GROUP BY
5.	… OVER (PARTITION BY…) – выполнение операций связанных с оконными функциями
6.	SELECT (Проекция «выживших» данных, рождение новых имен, математические операции, трансформация, агрегации, оконные функции)
7.	ORDER BY (Упорядочивание)
/Мастер: Задача приказа — сделать так, чтобы до финальной стадии «скульптуры» (SELECT) дошло как можно меньше «камня».
 
ФИЛОСОФИЯ SQL

Визуализируем процесс работы SQL: приказом, мы хирургически срезаем исходную таблицу до того вида, который нужен нам, выполняем некоторые операции (будь то математические операции или операции по трансформации) и выводим оставшуюся – «выжившую» часть в виде искусственной проекции, не вмешиваясь в исходник («из камня в скульптуру»).
При этом, работа под капотом следующая: приказом мы «запускаем» поток данных, исключаем лишнее нашими действиями, с целью получения точечного результата. Исключение лишнего должно происходить как можно раньше по цепочке выполнения – так, мы стремимся в кратчайшие сроки и не перегружая систему, получить целевой результат.
«SQL — это искусство исключения. Мы не строим отчет, мы исключаем лишние данные, чтобы осталась только истина»
 
Глава 2. БЛОК SELECT 
2.1 ВЫВОД НА ЭКРАН

Инструмент №1: Вывод данных на экран – [SELECT …]

SELECT не просто «показывает» данные, он создает виртуальную проекцию. Он берет отфильтрованный и сгруппированный поток строк и конструирует из них финальные столбцы отчета.
•	Исходная таблица в базе остается нетронутой.
•	Все элементы (колонки, агрегаты, расчеты, окна, подзапросы) перечисляются через запятую (после последнего запятая не ставится).
•	CTE (цеха) объявляются до блока SELECT.

2.2 БАЗОВАЯ МАТЕМАТИКА

- Любая математическая операция обязана быть названа при помощи алиаса AS - *грамотность

Инструмент №2: Математика в select - базовая – [SELECT … * …]

Математика в SELECT — это вычисление на месте. 
Математика не меняет данные в самом хранилище, она берет вводные данные, выполняет базовые математически операции (введеные оператором) и выдает результат «виртуально» – в отдельно созданной колонке, математика выполняется исключительно в рамках одной строки (горизонтали):

1. Базовые математические операции (+, -, *, /) между колонок, работают только между столбцами с числовым типом данных. Если типы данных разные — нужен принудительный перевод через инструмент CAST.
2. Базовые математические операции (+, -, *, /) между колонками и «вводимым числом» (константой). [пример: SELECT col*100 AS умножение].
3. Базовые математические операции (+, -, *, /) между ручным вводом/ колонками/ результатом агрегаторов. Запрещено напрямую смешивать в одной математической формуле сырую колонку и результат агрегатора. Если в формуле есть агрегат — сырая колонка обязана быть либо в блоке GROUP BY, либо сама обернута в агрегат, либо расчет должен идти через оконную функцию (OVER (PARTITION…) – которая физически вид не меняет.
4. Приоритет и скобки – стандартная математика в контексте приоритетов: сначала умножение и деление, потом сложение и вычитание. Нужно изменить порядок —используем скобки.

2.3 МАТЕМАТИКА – ПУСТОТА В СЛОЖЕНИИ И ВЫЧИТАНИИ

Инструмент №3: Предохранитель пустоты (NULL) в сложении и вычитании – маневр с COALESCE.

NULL в SQL – это пустота.  Любая математика с пустотой обнуляет (стирает) весь результат.
[100 + пустота (NULL) = пустота (NULL)]

Чтобы не потерять данные, используем оператор COALESCE (col, 0) 
Этот оператор подставит вместо пустоты – 0, тогда расчет не сломается.

Синтаксис: 
SELECT COALESCE (col, 0)
FROM tab1

//COALESCE превращает все найденные пустоты в указанной колонке – (col), в заданное нами значение, в данном примере заменяем пустоту на цифру ноль.
//Расшифровка синтаксиса: оператор COALESCE, «приклеивается» к указанной колонке и заменяет все найденные пустоты, на нужное нам значение, значение строго должно соответствовать формату данных в колонке.

2.4 МАТЕМАТИКА – НОЛЬ В ДЕЛЕНИИ

Инструмент №4: Защита от деления на ноль – NULLIF.

Без NULLIF один ноль в миллионе строк убьет весь отчет ошибкой «деление на ноль».
NULLIF можно назвать «игнорщиком» - по механизму, он будет игнорировать заданное значение, воспринимая это значение как пустоту. (пустота при делении безопасна).
Но по-научному, NULLIF это оператор сравнения на равенство. 

Его роль в математике — превращать опасное «число ноль» в «безопасную пустоту» и только строго при делении, при умножении используем COALESCE(col, 0).
Синтаксис:
SELECT 100/NULLIF(col,0) AS Колонка_с_делением
FROM tab1

Мы говорим системе: Сравни значение в заданной колонке с указанным нами значением (в нашем случае «0»,) -
*однако NULLIF позволяет указывает и другие значения, в том числе не числовые. 
- Если NULLIF находит значение в колонке, которое равно указанному нами значению — он вернет NULL (пустоту). Если они не равны — вернет исходное значение из колонки. 
Глава 3. БЛОК ТРАНСФОРМАЦИЯ
3.1 ЗАПЛАТКА ПУСТОТЫ – «КОАЛА»

Инструмент №5: Перехватчик пустот — [COALESCE]

COALESCE создает новую колонку для вывода результата выполнения своей ключевой роли (перехватчик пустот), взяв за основу указанную колонку (дублирует ее).
Выполняя свою роль, «оператор» движется по столбцу, заменяя все пустоты на новое значение – заплатку.
Иначе, COALESCE — это «заплатка» на дыру. Если система видит пустоту (NULL), она мгновенно меняет её на то, что мы указали в скобках. Принцип: «Если пусто — пиши вот это».

Синтаксис: 
SELECT COALESCE(col, 'замена') AS новая_колонка_с_заменой
FROM tab1

Если в колонке есть данные — система показывает их без изменений.
Если там NULL (пусто) — система подставит 'замену'.
Закон типов: Тип заплатки обязан соответствовать типу данных колонки. В числовую колонку ставим число, в текстовую — текст. Иначе запрос упадет.
 
Глава 3. ТРАНСФОРМАЦИЯ 
3.2 ЗАПЛАТКА ПУСТОТ В СТРОКЕ:
ОПЕРАЦИЯ «КОАЛЫ»

Сценарий существует, если выполняется условие №1 и №2.
№ 1. Если таблица содержит однотипные колонки по смыслу, например:
[факт, проект, норма]
№2. И наша задача заполнить пробелы хоть чем-то.

Синтаксис:
SELECT COALESCE (колонка1, колонка2, колонка3, ‘нет_данных’)
FROM tab1

/Так мы говорим системе проверь колонку 1 на пустоту, если встретил, посмотри в колонку 2, если и там пусто, иди в колонку 3, если и там пусто, ставь мое значение ‘нет_данных’.

COALESCE создает отдельную колонку для вывода результата своей работы. 
Суть - выполняя операцию «КОАЛЫ», система движется «слева направо» внутри одной строки (горизонтали) в рамках заданных столбцов, она ищет первую не-пустоту в рамках одного ряда, прежде чем перейти к следующей строке.

3.3 МАТЕМАТИКА - МОЩНАЯ АРИФМЕТИКА

Инструмент №6: Округление – [ROUND]
ROUND (col, n) — округление до n знаков после запятой; 
Синтаксис:
SELECT ROUND(col, n)
FROM tab1
Важно зафиксировать, что n может быть не только положительным.
•	ROUND(col, 2) — до сотых (стандарт для денег).
•	ROUND(col, 0) — до целого числа.
•	ROUND(col, -1) — округление до десятков (например, 123 превратится в 120).
/Математический закон: SQL округляет по классическому правилу: 0.5 и выше — в большую сторону, ниже — в меньшую.

3.3 МАТЕМАТИКА - МОЩНАЯ АРИФМЕТИКА


Инструмент №9: Смена числовых типов данных
[CAST(col as new type)], [col::new type]
Применяется, когда цифра застряла в текстовом формате, когда нужно сменить точность расчетов или отрубить дробный хвост.
Существует два равноправных синтаксиса смены типа («переплавка»):

Классический: CAST(col AS тип_данных) — стандарт СУБД.
Скоростной (Переплавка на ходу): col::тип_данных — Postgres-стандарт, пишется мгновенно на автопилоте рук.







Шпаргалка по типам («Во что плавим»):

•	col::INTEGER (или ::INT) — превращает в целое число, отрубая дробный хвост без округления (например, 12.99::INT превратится в 12).

•	col::NUMERIC(10,2) (или ::DECIMAL) — превращает в сверхточное число, где 10 — это общая длина числа, а 2 — знаки после запятой (10 и 2 – это стандарт).

•	col::TEXT (или ::VARCHAR) — плавим «нечто» в текст (нужно, если «нечто» надо склеить с каким-то словом).

 
ТРИ ОСНОВНЫХ МАТЕМАТИЧЕСКИХ ФОРМАТА
(КАРТА ВСЕЛЕННОЙ ЧИСЕЛ)

I. ЦЕЛЫЕ ЧИСЛА — INT / INTEGER
(INT / INTEGER — полные синонимы. INT сокращение от INTEGER.)
Суть: Числа без дробной части (1, 25, 1000). Занимают минимум места в RAM.
Закон округления: Переплавка в INT в PostgreSQL НЕ округляет число по правилам математики, а просто обрубает дробную часть.

II. ТОЧНЫЕ ЧИСЛА (ДЕНЬГИ / СКЛАД) — DECIMAL / NUMERIC
(DECIMAL / NUMERIC Полные близнецы по стандарту SQL ANSI, но numeric в Postgres встречается чаще)
Оба типа созданы для строжайшего коммерческого учета, где округление должно быть предсказуемым, а копейки не имеют права потеряться.
Суть: Числа с фиксированной, гарантированной точностью. 
Синтаксис: col::decimal(10,2) или col::numeric(10,2)
Параметры: 10 — всего цифр в числе (мощность), 2 — сколько из них выделено под дробный хвост (копейки).

III. ПЛАВАЮЩИЕ ЧИСЛА — FLOAT / REAL
Синтаксис на ходу: col::float
В чем разница FLOAT / REAL: разница физическая (в емкости «бака» для цифр).
Суть: Это часть приблизительных вычислений. Числа хранятся в двоичном формате, из-за чего компьютер может искажать сотые доли (например, вместо 0.5 физически записать 0.499999).
•	REAL — это легкий бак (4 байта). Хранит до 6–7 значащих цифр.
•	FLOAT (или DOUBLE PRECISION) — это тяжелый бак (8 байт). Хранит до 15 значащих цифр.
FLOAT категорически запрещен в финансовых отчетах.
 
Инструмент №10 «Топор» — [TRUNC]
Нужен тогда, когда математическое округление запрещено бизнес-логикой, и нужно обрубить лишний хвост, не меняя число до запятой.

Синтаксис: 
TRUNC(колонка, количество_знаков_после_запятой)

Физика работы: 
TRUNC(9.9, 0) – гарантированно выдаст 9 (топор просто срубил .9).
TRUNC(15.789, 2) – выдаст 15.78 (топор рубит тысячную долю 9, оставив сотые нетронутыми).

«Топор» вызывает ошибку, если применяется с типом (FLOAT).

Скоростной col::new type (для быстрой переплавки на ходу)
 
Глава 3. ТРАНСФОРМАЦИЯ
3.4 СМЕНА ТИПА ДАННЫХ И РАЗВЕДКА

ДОСТУПНЫЕ НЕЧИСЛОВЫХ ФОРМАТЫ

VARCHAR / TEXT — Обычные текстовые буквы и символы.
DATE — Чистая дата: только день, месяц и год (2026-06-05). Без часов и минут.
TIMESTAMP — Дата со временем: день, и точное время до секунд (2026-06-05 20:45:00).
Скоростной маневр по ликвидации времени в timestamp: col::date — и система мгновенно отрубает часы, минуты и секунды, оставляя только чистый день.

ТРИ БОЕВЫХ СЦЕНАРИЯ (РАСКРЫТИЕ РОЛЕЙ)

1.	Текст - в Дату / Дата со временем: 
CAST(text_column AS DATE) или CAST(text_column AS TIMESTAMP)
Зачем: Когда даты в формате текста.

2.	Число - в Текст: CAST(id_column AS VARCHAR)
Зачем: СУБД физически запрещает склеивать числа с текстом в функциях сборки текста. (будущий инструмент CONCAT).

3.	Дата в Текст: CAST(date_column AS VARCHAR)
Зачем: Когда чистую календарную дату (2026-06-05) нужно разобрать как обычное текстовое слово, чтобы применить к ней строковые фильтры.
 
Глава 3. ТРАНСФОРМАЦИЯ
3.4 СМЕНА ТИПА ДАННЫХ И РАЗВЕДКА

Инструмент №12: Разведки качества данных – [CAST]
CAST — это еще и проверка на вшивость. Если при попытке перевести данные одной колонии в какой-либо формат, к примеру CAST(col AS INT) – запрос падает, значит, в данных есть «мусор» (буквы там, где должны быть только цифры).
 
Глава 4. ПОДЗАПРОС 
4.1 ВЛОЖЕННЫЙ (SELECT) В SELECT – НЕ АКТУАЛЬНО
(скалярный подзапрос – инструмент «сверхпозиции»)

Инструмент № 13: Сравнение с «мировой константой» – [(SELECT) в SELECT]

Располагая запрос (SELECT …) внутри основного блока SELECT, мы принудительно формируем отдельную колонку, где содержится одно единственное значение для всех строк (курс валют, план на месяц, среднее по больнице).
•	Жесткий закон «Правило 1х1»: Подзапрос внутри SELECT обязан возвращать строго одну колонку и одну строку (ячейку 1х1). Иначе система упадет с критической ошибкой.
•	Защита Мастера: Чтобы «Правило 1х1» не нарушалось, внутри подзапроса всегда используют агрегаторы (AVG, MAX) или LIMIT 1.

КРИТИЧЕСКИЙ СИСТЕМНЫЙ ЗАПРЕТ (ОТК):
Этот инструмент — тяжелый и иррациональный. Если таблица огромная (от 10 000 строк), сервер будет выполнять этот внутренний подзапрос для каждой строки отдельно. Миллион строк = миллион скрытых циклов. Система задохнется.
ПРАВИЛО МАСТЕРА: столкнулся со сценарием «мировой константы» в реальном кейсе? Забудь про этот инструмент. Не пиши подзапрос в SELECT. Выноси этот расчет в изолированный CTE цех (WITH), чтобы вычислить константу один раз и не перегружать процессор СУБД. – Применение инструмента идет далее.
 
Глава 4. ПОДЗАПРОС
4.2 АЛЬТЕРНАТИВА – МАНЕВР «ШТАМП» (CROSS JOIN)

Инструмент №14: «Мировой константа» через CTE цех
Безопасная и быстрая замена тяжелому подзапросу в SELECT. Вместо того чтобы мучить процессор скрытыми циклами для каждой строки, мы вычисляем константу ОДИН раз в изолированном цеху (виртуальная таблица), а затем приклеиваем её на весь отчет.

Синтаксис:
WITH cte_constant AS (
    SELECT AVG(col) AS константа FROM tab1 -- Посчитали 1 раз
)

SELECT 
    t.col_name,
    c.константа -- Наш штамп доступен в каждом ряду
FROM tab1 AS t
CROSS JOIN cte_constant AS c -- Приклеили клей-моментом

Физика работы (Маневр «Штамп»):
CROSS JOIN — Единственный вид склейки таблиц, которому НЕ нужны общие ID или ключи (ON t.id = c.id).
Система берет основную таблицу (миллион строк) и нашу ячейку-константу из CTE, а затем просто штампует это значение на каждую строчку сверху вниз.
ПЛЮС ДЛЯ СИСТЕМЫ: Вычисление происходит единожды. Скорость работы отчета вырастает в сотни раз.










Глава 4. ПОДЗАПРОС 
4.3 ВЛОЖЕННЫЙ SELECT В БЛОКЕ WHERE

Инструмент №15: Создание вложенного условия – [(SELECT) в WHERE]

Подзапрос в WHERE или создание «вложенного условия», существует там, где для вывода данных в отчете, критически важны уточнения, в таком сценарии, таблица главного запроса обращается сама к себе [I], либо к другой таблице [II].

I. В рамках одной таблицы;
/Подзапрос таблицы к себе самой — это легальный способ найти, например, «всех, кто выше среднего» в рамках этой самой таблицы – именно так решаются задачи типа «найди поставщиков, чей объем выше среднего по всей таблице».
II. Когда уточняем у другой таблицы:
Мы «дистанционно» связываемся со второй таблицей, где обязана существовать идентичная колонка из первой таблицы.

Сам формат подзапроса делится на два режима работы:
I. Точечный режим (=, >, <) 
В данном режиме мы жестко ограничены в написании вложенного SELECT, наш запрос внутри должен выводить только одну ячейку (1x1), по которой будет выполняться условия фильтра.
/Мы не можем проводить точечную операцию с массивом, поэтому внутри вложенного select, нельзя пытаться получить массив. (MAX, MIN, AVG, LIMIT – помощники в данном режиме).
II. Режим множества, IN
В данном режиме, жесткие ограничения только в количестве колонок, а именно – одна колонка: (1 x N) полученный результат во вложенном SELECT обязан выводить только одну колонку, но владеть списком строк  – в этом кроется основная суть IN и вложенного SELECT из WHERE – сравнение исходного списка главного запроса и полученного списка, прошедшего условия фильтра в подзапросе, с последующим исключением тех, кто выбыл в подзапросе.
Синтаксис:
SELECT список имён
FROM таблица 1
WHERE КОЛОНКА_ГОНЕЦ IN (SELECT КОЛОНКА_ГОНЕЦ FROM таблица 2.) 
Создавая вложенный запрос через IN, мы буквально срезаем весь список таблицы, оставляя только то, что совпало с тем списком, который был порезан внутри условия вложенного select.
Оба режима могут существовать в обоих сценариях: 
Сценарий 1: В рамках одной таблицы, во вложенном select, в блоке FROM будет исходная таблица., 
Сценарий 2: В рамках общения двух таблиц – во вложенном select, в блоке FROM будет другая таблица., главное условие выполнения сценария IN и двух таблиц: колонка/гонец должна быть и в первой, и во второй таблице.
[IN отправляет список «объектов» на испытание во вложенный запрос, те кто вернулся – стали новой основой для ГЛАВНОГО select запроса]

Ловушка NULL: Если подзапрос вернет список, в котором есть хотя бы один NULL [пустота], весь внешний запрос может вернуть пустоту. Для безопасности в IN подзапросах всегда следует отсекать пустые значения через WHERE col IS NOT NULL. 
Этот стоп-кран особенно критичен для оператора NOT IN. В обычном IN (поиск совпадений) одиночный NULL в списке просто проигнорируется системой, но в NOT IN (поиск исключений) один-единственный NULL полностью уничтожит всю выборку. 

1. Когда подзапрос в WHERE НЕ актуален (Режим множества IN)
WHERE id IN (SELECT id FROM другая_таблица WHERE ...)
Этот маневр устарел. Гораздо проще, быстрее и нагляднее перенести эту логику в CTE-цех: заранее отфильтровать вторую таблицу, а потом просто склеить их через обычный INNER JOIN. Система сама отсечет всё лишнее («выживут» только те, кто совпал.
2. Когда подзапрос в WHERE актуален (Точечный режим)
Представим задачу: «Найти сотрудников, у которых зарплата выше средней по больнице».
Если писать через подзапрос в WHERE, это выглядит как мгновенный точечный выстрел:
WHERE salary > (SELECT AVG(salary) FROM employees)
Почему это работает круто: Внутренний подзапрос возвращает всего одну ячейку (1х1). Планировщик PostgreSQL видит, что этот подзапрос независим. База данных вычисляет эту среднюю зарплату ровно один раз, превращает её в константу и дальше просто фильтрует WHERE salary > 50000.

Глава 5. БЛОК SELECT: AS (АЛИАС)

Инструмент №16: присвоение своего имени – [AS]
Оператор AS 
– располагается в select, и присваивает имя колонке, математической операции или операции по трансформации, – после которой установлен;
– или в блоке from, задавая новое имя целой таблицы – после которой установлен;

Система не учитывает новое название колонки в блоке WHERE, так как он выполняется первее, но учитывает новое имя таблицы, так как FROM это точка старта.

Имя в AS пишется без кавычек, но если внутри имени нужен пробел (например, Общая сумма), то все имя обязательно зажимается в двойные кавычки "Общая сумма".
 
Глава 6. БЛОК SELECT: ЯРЛЫКИ (CASE)

Инструмент № 17: Присвоение ярлыков на базе условий IF-ELSE – [CASE]
CASE располагается в блоке SELECT. CASE создает отдельную колонку (помним про AS), и записывает в соответствующие строки те «значения», которые будут соответствовать заданным нами условиям. 
/если в «заданной» колонке найдешь цвет «красный» - присвой имя ‘опасно’, если ‘зеленый’ – ‘безопасно’, иначе – ‘не определено’

Синтаксис:
SELECT ТРУБА, 
CASE
WHEN ЦВЕТ=’СИНИЙ’ THEN ‘ТЕХНОЛОГИЧЕСКАЯ ТРУБА’
WHEN ЦВЕТ=’КРАСНЫЙ’ THEN ‘ТРУБА ДЛЯ ТЕПЛО СЕТЕЙ’
ELSE ‘ТРУБА_НЕ_ОПРЕДЕЛЕНА’
END AS НОВОЕ_ИМЯ_КОЛОНИК_В_НАШЕМ_СЛУЧАЕ_ПУСТЬ_БУДЕТ_«ПРИНАДЛЕЖНОСТЬ»
FROM ТАБЛИЦА_С_ТРУБАМИ
WHERE ДИАМЕТР_ТРУБЫ IN (89, 108, 219)
     
/ Типы данных: Все результаты после THEN и ELSE в рамках одного блока CASE, обязаны быть одного типа (либо все — 'текст', либо все — числа). Нельзя в одной колонке смешивать форматы.
Ключевые детали сборки:
1.	WHEN: Условие к системе;
2.	THEN: Если условие выполняется, система вешает ярлык с соответствующим значением (вводим «что хотим видеть» после then) – система не ограничивает нас в количестве строк THEN.
3.	ELSE: Запасной вариант. Если ни одно WHEN не сработало, запишется это значение. Если ELSE мы не указали — система запишет NULL [пустоту].
4.	END: «Замок», закрывающий логический блок.
5.	AS: Обязательное имя для колонки-результата.
/Внутри WHEN можно использовать и сложные логические связки через AND / OR (например, WHEN цвет = 'красный' AND статус = 'активен' THEN...).
Исходя из примера, мы видим, что в блоке CASE мы вольны прикреплять ярлыки «в фоновом процессе», по тем колонкам, не участвующим напрямую в запросе. 
Глава 7. КОМАНДА ЮВЕЛИРОВ: 
ВЫСШИЙ УРОВЕНЬ ОЧИСТКИ ДАННЫХ ОТ МУСОРА И КОРРЕКТНОГО ВЫПОЛНЕНИЯ JOIN, WHERE, GROUP BY, А ТАКЖЕ ФУНКЦИЙ АГРЕГАЦИЙ
Все команды ювелиры являются обязательным инструментом мастера при работе с «грязными» таблицами, только благодаря данным командам мы способны очистить таблицу от мусора, который гарантированно влияет на точность вывода данных, проведения расчетов, выполнения условий фильтрации, создания групп и сшивку миров, без них любой запрос неизбежно будет содержать ошибки, с ними и только с ними мы способны буквально спасти миллионы.

Инструмент №18: «CUT» – значит отрезать – [LEFT/RIGHT]
Мы можем создать колонку, в которой от любого значения из указанной колонки, останется n-количество символов от левого или правого конца:
/SELEСT LEFT(name_col, n)
где LEFT – это сторона, с которой будем «отрезать», в скобках указывается колонка, данные которой будем резать, и n – это количество символов, которые система ОСТАВИТ от соответствующего начала.
Как и любая трансформация, CUT (* авторское название) требует AS, иначе заголовок колонки будет техническим и нечитаемым.
Если в ячейке попадется NULL, функции LEFT/RIGHT не упадут в ошибку, они просто вернут NULL.

Инструмент №19: Смена регистра – [UPPER/LOWER]
/SELECT UPPER(name_col) или SELECT LOWER(name_col)
Трансформация, переводит все «значения» указанной колонки в ВЕРХНИЙ или нижний регистр. Требует AS.
/Если сделать GROUP BY без смены регистра, система создаст две разные группы для одного и того же материала.
/Чтобы фильтр в WHERE или группировка в GROUP BY сработали идеально, всегда «пропускай» колонку через LOWER или UPPER. Так ты гарантируешь, что ни одна строка не потеряется из-за разного написания.
/При поиске конкретного слова в блоке WHERE приводи к регистру и колонку, и поисковый запрос (например: WHERE LOWER(material) = 'цемент').

(UPPER/LOWER) ОБЯЗАТЕЛЬНО ПРИМЕНЯЕТСЯ В WHERE, GROUP BY, при JOIN в связке с «Тримером» (TRIM).

Инструмент №20: Удаление пробелов – «Триммер» – [TRIM]
Без TRIM любой идеальный JOIN или WHERE просто промахнутся мимо цели.
TRIM — это нож для пробелов, который срезает все паразитные (лишние) пробелы в начале и в конце строки, оставляя только чистое смысловое ядро.
Синтаксис базовый: 
SELECT TRIM(column)
Система сканирует строку с обоих концов. Она видит пробел впереди и пробел позади, а затем «откусывает» их. 
Пробелы между словами внутри строки TRIM никогда не трогает — они защищены.
TRIM работает строго с текстовыми типами данных (VARCHAR, TEXT)

Сценарии применения (Где этот болт держит конструкцию)
Сценарий 1: Броня в блоке WHERE – чтобы фильтр сработал на 100%
Сценарий 2: Спасение мостов в блоке JOIN (сейв от раздува)
/Дешифровка: Одновременно опускаешь регистр букв до маленьких (LOWER(t1.col)) и срезаешь невидимые пробелы с обеих сторон (TRIM(t1.col) в обеих таблицах. Мост становится неуязвим.
… INNER JOIN tab1 ON (LOWER(TRIM(tab1.k1)) = (LOWER(TRIM(tab2.k1))…
Сценарий 3: Унификация группировки – чтобы одноименные строки гарантированно слиплись в одно целое

Синтаксис специализированный (встречается реже):
•	LTRIM(column) — режет пробелы только слева 
•	RTRIM(column) — режет пробелы только справа

 
Инструмент №21: СЦЕПКА – [CONCAT]
CONCAT(name_col1, ‘ ‘, name_col2) – создает отдельную колонку, в которой сцепляет данные из разных столбцов одной горизонтали в одно целое значение, чтобы поставить пробел – его следует заключить в одинарные кавычки.
/ Типы данных: важно помнить, что CONCAT работает с текстом. Если нужно приклеить к тексту число (например, ID заказа), его сначала нужно «расплавить» через CAST(id AS VARCHAR/TEXT)
/CONCAT_WS(' ', col1, col2, col3) — продвинутая сцепка с разделителем, где пробел указывается один раз в начале, а не пишется между каждой колонкой.
 
Глава 8. БЛОК FROM: ТОЧКА ВХОДА И МАТЕРИАЛИЗАЦИЯ ПРОСТРАНСТВА

Инструмент №22: Адрес хранилища данных – [FROM]
Логика: FROM — это приказ системе определить физический адрес данных. 
Таблицы в базе — это изолированные острова, и через FROM мы указываем, на какой конкретно остров высаживается «поисковый отряд». 
Синтаксис (Базовый): 
SELECT col FROM table
Синтаксис (AS архитектурный): 
FROM very_long_name AS t1
Закон Первородства: несмотря на то, что в коде, FROM мы пишем на второй строке (после SELECT), система под капотом выполняет его первым. Это абсолютное начало координат запроса, где присвоенный алиас автоматически становится «законом» для всех остальных блоков: и для SELECT, и для WHERE, и для GROUP BY.
1.	Архитектурный Алиас (AS): Присвоение короткого «позывного» таблице прямо в блоке FROM.
o	Как только мы прописали FROM orders AS t1, старое длинное имя orders перестает существовать для системы в рамках этого запроса.
2.	Точечная адресация (Префиксы): Алиас во FROM обязывает обращаться к колонкам через точку: t1.status, t1.price. Это исключает «амбивалентность» (двусмысленность) для сервера, когда в запросе появится больше одной таблицы.




 
Глава 8. БЛОК FROM, ПРОИЗВОДНАЯ ТАБЛИЦА

Инструмент №23: Чистая таблица, [FROM (SELECT) AS] 
– НЕ ВСЕГДА АКТУАЛЬНО
FROM (SELECT) AS – «Создание временной, предварительно обработанной таблицы прямо в воздухе».
По умолчанию FROM берет «грязную» таблицу из базы, где не всегда, для анализа, нужны и 50% всего массива данных.
Чтобы «отрезать гнилой кусок пирога», до начала хирургической операции по извлечению данных, система позволяет из обычного запроса SELECT, сделать чистую основу для ГЛАВНОГО SELECT запроса. 
Для этого, диктуем обычный запрос: 
SELECT city, AVG(salary) AS average_salary 
FROM workers
GROUP BY city
Прячем его в скобки, подкладываем во FROM ГЛАВНОГО SELECT запроса, и обязательно присваиваем имя, иначе сервер заблокирует запрос с ошибкой (таблицы не существует)/
Полный синтаксис:
SELECT t1.city, t1.average_salary
FROM (
    SELECT city, AVG(salary) AS average_salary
    FROM workers
    GROUP BY city
) AS t1
WHERE t1.average_salary > 100000;
/Для внешнего (главного) запроса, подзапрос из FROM выглядит как самая обычная чистая таблица.
Сценарии применения (Когда без этого не обойтись):
Двухуровневая агрегация – проще говоря, нужен тогда, когда с первым результатом какой-либо агрегации нужно выполнить еще одну функцию агрегации.
Будущая АКТУАЛЬНАЯ альтернатива – CTE цеха которые делают код чище и читаемые, избавляя от громоздких вложенных скобок во FROM (их мы разберем дальше).

 
Глава 8. БЛОК FROM, МОСТ МЕЖДУ ПРОИЗВОДНЫМИ ТАБЛИЦАМИ

Инструмент №24: Соединение двух чистых таблиц и их связка
Соединение двух независимых подзапросов из FROM в один отчет - стыковка двух виртуальных вселенных, созданных на лету.
В аналитике это применяется постоянно, когда нужно объединить данные из двух тяжелых таблиц, но ты не хочешь тащить в мост (JOIN) миллионы сырых строк. Ты сначала «сушишь» каждую таблицу отдельно в свой «чистый пирог» во FROM, а потом соединяешь эти два чистых результата.
Архитектурный чертеж (Синтаксис стыковки двух подзапросов)
SELECT
    t1.city,
    t1.total_sales,
    t2.total_workers
FROM (
    -- Виртуальный остров №1: Выручка по городам
    SELECT city, SUM(sales) AS total_sales
    FROM orders
    GROUP BY city
) AS t1
JOIN (
    -- Виртуальный остров №2: Штат по городам
    SELECT city, COUNT(id) AS total_workers
    FROM staff
    GROUP BY city
) AS t2 
ON t1.city = t2.city;
Точка опоры (ON): Колонки, по которым идет сцепка, должны быть явно выведены внутри обоих подзапросов.

НЕ ВСЕГДА АКТУАЛЬНО, данная функция полностью укладывается на связку двух очищенных таблиц в CTE цеху.




Глава 8. БЛОК FROM, ПРИСВОЕНИЕ ИМЕН

Инструмент №25: Системный позывной таблицы — [AS в блоке FROM, JOIN]
Логика: Когда мы вызываем таблицу из хранилища, мы вешаем на неё короткий ярлык (алиас). Это нужно для того, чтобы во всех остальных частях приказа (SELECT, WHERE, JOIN) не писать длинное «имя» таблицы, а обращаться к ней по короткому позывному.
Синтаксис: FROM very_long_table_name AS t1
1.	Закон Приоритета: AS в блоке FROM — это «первичный кирпич». Он рождается в самом начале исполнения запроса, поэтому его видят абсолютно все блоки, стоящие выше или ниже.
2.	Диктатура имени: как только мы присвоили таблице имя (например, t1), старое длинное имя для системы перестает существовать. 

JOIN и AS.
Когда таблиц становится больше одной, синтаксис превращается в цепочку: каждая новая таблица заходит через приказ JOIN и сразу получает свой позывной. Синтаксис присвоения имен нескольким таблицам:
FROM table_leader AS t1
JOIN table_partner AS t2 ON t1.id = t2.parent_id
JOIN table_extra AS t3 ON t2.id = t3.partner_id
Разбор:
1.	Первый позывной: Задается в блоке FROM. Это «база», к которой мы всё приклеиваем.
2.	Последующие позывные: Задаются сразу после названия таблицы в каждом блоке JOIN.









Глава 9. БЛОК JOIN: 
СТЫКОВКА МИРОВ

Логика: Этот блок создает временный мост между таблицей-лидером (в блоке FROM) и таблицей-партнером, формируя одну единственную сверх-таблицу, в рамках которой мы продолжаем работу по отсеву. 
Стыковка происходит строго горизонтально: система берет строку из левого мира, ищет совпадение (или не ищет – зависит от вида стыковки) в правом мире и склеивает их в одну длинную супер-строку. По сути, мы можем входить в FROM с любой таблицы, связка произойдет, но таблица в FROM приобретает статус лидера, что важно для JOIN LEFT. 

Инструмент №26: Мост пар – [INNER JOIN ON]
Этот мост пропускает в итоговый отчет только тех, у кого есть стопроцентная взаимность (пара). Если строка из левой таблицы не нашла себе пару в правой (или наоборот) — она уничтожается и полностью исчезает из финального отчета.
Синтаксис:
SELECT o.id, o.price, c.client_name
FROM orders AS o
INNER JOIN clients AS c ON o.client_id = c.id
INNER – «Оставь только те строки, которые нашли свою пару»
ON – Точка опоры: приказ, который задает условие сцепки. Если значение из таблицы-лидера, нашло свою копию в таблице-партнёре, связь между строками двух таблицу существует, и формирует сверх-строку в рамках сформированной сверх-таблицы.
*Сверх-таблица –это объединение таблиц в одну большую.
/Ключевое слово INNER можно опускать (просто написать JOIN), СУБД по умолчанию воспримет это как INNER JOIN.
 
Глава 9. БЛОК JOIN: 
НЕУБЕВАЕМАЯ ТАБЛИЦА ЛИДЕР

Инструмент №27: LEFT JOIN — Левый мост (Удержание контроля).
Логика: Самый востребованный инструмент аналитика. Этот мост защищает левую таблицу (лидера) от любых потерь. Все строки из таблицы во FROM гарантированно выживут и попадут в отчет. Если строка из правого мира не нашла пару — мост все равно строится, но на месте правых данных система оставляет безопасную пустоту (NULL).
Синтаксис:
SELECT c.client_name, o.id AS order_id
FROM clients AS c
LEFT JOIN orders AS o ON c.id = o.client_id;
Механика NULL: Если клиент «Иванов» ничего не купил, он все равно останется в отчете. Просто в колонке order_id напротив него будет написано NULL.
Смысл: Идеально для поиска просадок и долгов. Например: взять список всех плановых задач (LEFT JOIN факт выполнения) -> где факт IS NULL -> мгновенно получили список сорванных этапов.
Капкан №1: Раздувание строк: JOIN осуществляет сцепку «строка к строке». Если в правой таблице по точке опоры (ON) найдется не одна строка, а несколько, система продублирует левую строку под каждую правую. Отчет раздуется, а все будущие расчеты из агрегаторов увеличатся в разы. Поэтому точка опоры в правой таблице *в большинстве случаев, должна быть уникальным ключом (PK ID) или уникальным составным ключом, сформированным через AND в блоке JOIN.
Исключение (Сценарий «Один-ко-многим»): Размножение строк — это норма, если нужна детализация (например: один Договор -> LEFT JOIN пять Этапов оплат). Главное — помнить о размножении при дальнейших расчетах.
Капкан №2: Фильтр по второй таблице (таблица-партнёр) превращает LEFT в INNER, тем самым теряя свой первоначальный замысел сохранения таблицы-лидера. Чтобы этого не допустить, применяем щиты защиты (Как не сломать мост):
1.	Маневр через NULL в WHERE – в фильтр по второй таблице мы вносим явное условие (если пустота)
WHERE o.status = 'Оплачено' OR o.status IS NULL
2.	Маневр через AND в блоке ON – мы задаем условия фильтра на этапе стыковки:
LEFT JOIN orders AS o ON c.id = o.client_id AND o.status = 'Оплачено'
 
Глава 9. БЛОК JOIN: 
НЕЯВНЫЕ СВЯЗИ

Инструмент №28: Создание уникального ID вручную – [JOIN ON (AND)]
В IT этот маневр называется составным соединением (Composite Join) или неявным маппингом. Задача здесь — найти несколько общих точек опоры в разных таблицах, связать их вместе и заставить систему поверить, что эта комбинация и есть уникальный ID.
*Глубинный смысл: Стыковка по смысловым координатам
Представь две таблицы из ПТО. Первая — «Снабжение» (где ведут учет прихода материалов), вторая — «Сметы» (где заложены лимиты). Программисты не заложили в базу сквозной material_id. Всё, что у нас есть — это текстовые названия и характеристики.
Если сделать JOIN только по названию материала (ON t1.name = t2.name), мост рухнет в хаос дубликатов, потому что слово «Цемент» или «Арматура» встречается в таблицах сотни раз для разных марок и объектов.
Решение: Мы строим мост не по одной опоре, а по целой системе опорных координат. Мы говорим системе: «Строка левого мира равна строке правого мира только тогда, когда совпадает – И материал, И его марка, И объект строительства».
/Создание уникального ID кустарным методом
Архитектурный чертеж (Синтаксис Composite JOIN) 
SELECT 
    s.object_name,
    s.material_name,
    s.material_grade,
    s.quantity_fact,
    e.quantity_estimate
FROM supply AS s
LEFT JOIN estimates AS e 
    ON  s.object_name   = e.object_name 
    AND s.material_name = e.material_name 
    AND s.material_grade = e.material_grade;
Оператор AND в блоке ON – Это главный замок. Он заставляет систему проверять все условия одновременно. Строка справа приклеится только если совпадут все три параметра.
Общая альтернатива: Сборка кустарных текстовых ключей через CONCAT внутри CTE перед соединением.
 
Глава 9. БЛОК JOIN:
НЕЯВНЫЕ СВЯЗИ – КАПКАНЫ

Главные капканы сценария JOIN без ID 
Капкан 1: Невидимые пробелы и разный регистр
Текст — коварная материя. Если в первой таблице написано 'ЖК Центр', а во второй 'ЖК Центр ' (с невидимым пробелом на конце) или 'жк центр' (с маленькой буквы), система скажет: «Не совпало» — и LEFT JOIN выкинет NULL.
•	Защита (Инструменты ювелира в блоке ON): Мы зачищаем текст прямо в точке сцепки с помощью LOWER и функций удаления пробелов (TRIM):
ON TRIM(LOWER(s.material_name)) = TRIM(LOWER(e.material_name)) – эталонный стандарт чистого маппинга.
Капкан 2: Размножение данных (Дубликаты)
Если даже при трех условиях в правой таблице найдутся две абсолютно одинаковые строки (например, две сметные строки на Цемент М500 для одного объекта), система размножит твою левую строку.
Синтаксическая дотошность:
В блоке ON можно использовать сколько угодно операторов AND, связывая текстовые, числовые колонки или даты. Главное — типы данных в каждой паре должны совпадать (текст сравниваем с текстом, число с числом).
Капкан 3: Пустоты
Чтобы составной мост не разбился о пустоту, текстовые NULL нужно перехватывать через «Коалу» прямо внутри условий ON:
AND COALESCE(s.material_grade, 'нет') = COALESCE(e.material_grade, 'нет')
Логика: Если с обеих сторон прилетят пустоты, COALESCE превратит их в одинаковое слово 'нет'. Условие 'нет' = 'нет' выдаст TRUE, замок закроется, и мост выстоит.

 
Глава 10. БЛОК WHERE: 
ФИЛЬТРАЦИЯ

Инструмент №29: Фильтрация [WHERE]. 
Самый ключевой и приоритетный блок.
WHERE «условия»
Через данный блок мы просеиваем данные, задавая им условия по выводу истины и исключения лишнего.
Помни: WHERE выполняется до SELECT, нельзя фильтровать в WHERE по новым именам колонок, которые создали в SELECT.
Инструментарий: В WHERE мы можем использовать простейшие математические операции: (=, >, <, <=, >=), а также мощные операторы: IN (список), BETWEEN (диапазон), LIKE (поиск по маске) и IS NULL (проверка на пустоту).


 
Глава 10. БЛОК WHERE: 
СОЧЕТАНИЕ УСЛОВИЙ

Инструмент №30: Перечисление условий, [AND]
[AND – сужает сито]
В блоке WHERE запятая синтаксически запрещена. Если нужно, чтобы запись прошла через фильтр по нескольким критериям одновременно, единственный путь — логическая связка условий при помощи оператора AND.
Синтаксис: 
WHERE условие_1 AND условие_2
Глубинный смысл: AND — это умножение вероятностей. Чтобы строка выжила, она должна соответствовать всем истинам одновременно.

Инструмент №31: Логическое ИЛИ в блоке where, [OR]
[OR – расширяет сито]
Если применяется только OR, диктуем системе «или выполняется условие_1 или выполняется условие 2»;
Если выполняется совместно с AND, расширяет выборку (в отличие от чистого AND, который её сужает).
Приоритет: В сложных запросах AND выполняется раньше, чем OR. Чтобы задать свой порядок, используем скобки.
Синтаксис: 
WHERE (условие_1 OR условие_2) AND условие_3.
Если много OR по одной и той же колонке – лучше применить IN (1, 2, 3). Это есть «скрытый OR», который пишется быстрее.
Маневр профи (Ловушка NULL): если нужно видеть пустые записи в отчете. Всегда добавляй OR колонка IS NULL;








Глава 10. БЛОК WHERE: 
ИНСТРУМЕНТАРИЙ В WHERE

Инструмент №32: Поиск по корню слова, [LIKE]
Оператор [LIKE] – это оператор «совпадения», применяется если нужно осуществить – «поиск по фрагменту/части слова», иначе можно сказать «поиск по КОРНЮ слова» с разными приставками и окончаниями. 
Синтаксис: 
WHERE КОЛОНКА_ДЛЯ_ПОИСКА LIKE ‘%КОРЕНЬ%‘

Пример:
WHERE ИСКОМЫЙ LIKE ‘%РОД%‘
В результате, система покажет из колонки ИСКОМЫЙ только то, что включает в себя «корень» РОД, например: РОДительсикй, РОДной, РОДина, предРОДовой и т.д.
Чтобы система игнорировала ПРИСТАВКУ, % (символ-джокер) ставится перед словом; 
Чтобы система игнорировала ОКОНЧАНИЕ, % ставится после корня.
Маневр профи (Регистр): Оператор LIKE чувствителен к регистру, 'Род' и 'род' — для него разные миры.
Решение:  используем ILIKE – он игнорирует высоту букв и находит 'РОД', 'род' и 'Род' одинаково успешно.
 
Глава 10. БЛОК WHERE: 
ИНСТРУМЕНТАРИЙ В WHERE

Инструмент №33: КПП – [оператор IN]
Оператор IN («ВКЛЮЧИТЬ В СПИСОК») — это «быстрый фильтр» для колонки, в которой нам нужно отобрать несколько конкретных строк. Вместо того чтобы писать много OR, мы создаем контрольный список.
Синтаксис:
WHERE колонка_с_данными IN ('значение_1', 'значение_2', 'значение_3')
Система пропустит только те строки (всю горизонталь), чье указанное имя совпало с контрольным списком указанных имен.

Если нужно исключить конкретный список – применяем NOT IN («ИСКЛЮЧИТЬ ИЗ СПИСКА»). 
Например, «Покажи все диаметры, кроме 57 и 89». Это моментально очищает отчет от ненужного шума.

Важно - одна пустота в NOT IN обнулит весь отчет. 
Как занулить этот капкан: 
Очистка списка от пустот (NULL) прямо в условиях – добавляя жесткое условие IS NOT NULL перед списком.
Пример:
WHERE diameter IS NOT NULL AND diameter NOT IN (57, 89) 
Логика: Мы сначала убираем из проверки все пустоты (IS NOT NULL), а затем спокойно прогоняем чистые числа через КПП. Запрос спасен.
 
Глава 10. БЛОК WHERE: 
ИНСТРУМЕНТАРИЙ В WHERE


Инструмент №34: Оператор фильтра «между» — [BETWEEN]
Оставь только значения ОТ и ДО. Корректно работает ТОЛЬКО с числами и датами, с текстом будут проблемы.
Главная особенность: Границы включены в результат. Если ищешь BETWEEN 10 AND 20, то и 10, и 20 попадут в отчет.
Синтаксис:
WHERE колонка_с_данными BETWEEN значение_1 AND значение_2
Применение в разных мирах:
1. Числа: WHERE price BETWEEN 100 AND 500 (всё, что стоит от 100 до 500 включительно).
2. Даты: WHERE date BETWEEN '2023-01-01' AND '2023-01-31' (весь январь).
Капкан TIMESTAMP (Потеря последнего дня):
Если колонка хранит дату со временем, BETWEEN отсечет всё, что произошло после 00:00:00 последнего дня. Чтобы отчет не терял данные за последние сутки, всегда принудительно сбивакм время у колонки перед проверкой:
WHERE date::date BETWEEN '2023-01-01' AND '2023-01-31

Если нужно найти внешние значения, используют NOT BETWEEN. 
Например: WHERE price NOT BETWEEN 10 AND 1000 — это мгновенный поиск слишком дешевых или неоправданно дорогих товаров.
 
Глава 10. БЛОК WHERE: 
IS NULL – ОХОТНИК ЗА ПУСТОТОЙ

В SQL NULL — это не ноль и не пробел, это «неизвестность» – пустота. Поэтому обычные знаки сравнения (=, <, >) с ним не работают. Чтобы поймать пустоту, нужен специальный «детектор».

Инструмент №35: Охотник за пустотой — [IS NULL]
IS NULL – это оператор для поиска строк, где данные не были введены (пустота).

Критическое правило: Никогда не пишут WHERE column = NULL. Это логическая ошибка, которая вернет «ничего». Правильно только IS NULL.

Два режима охоты IS NULL:
1. IS NULL (Поиск дыр) – Находит строки, где значение отсутствует.
Пример: WHERE email IS NULL (найти пользователей, которые не оставили почту).

2. IS NOT NULL (Поиск наполненности) – Находит только заполненные строки.
Пример: WHERE phone IS NOT NULL (выгрузить только тех, кому можно позвонить).
 
Глава 10. БЛОК WHERE: 
IS NULL – ОХОТНИК ЗА ПУСТОТОЙ

Инструмент №36: Масштаб пустот – [IS NULL + COUNT(*)]

Используем IS NULL вместе с COUNT(*), чтобы понять масштаб пустот в данных.
SELECT COUNT(*) FROM table WHERE column IS NULL 
— это быстрый замер «пустот» в конкретной колонке.

Обо что разбивается IS NULL при работе с текстом :
Если в ячейках сидят «голые» кавычки ('') или пробел ' ', то для сервера эта ячейка НЕ является NULL. Она заполнена символом пустоты. В итоге WHERE col IS NULL пролетит мимо этой строки и не поймает её. Чтобы гарантированно запереть этот капкан, всегда стоит применять страховку – триммер: 
WHERE (col IS NULL OR TRIM(col) = '')

 
Глава 11. БЛОК GROUP BY

Инструмент №37: Cоздание групп – [GROUP BY и одна колонка]
GROUP BY col
При помощи GROUP BY, несколько одноименных строк, из указанной колонки, сводятся в одну. Истинно работает в связке с функциями агрегации.
Жёсткое правило: если в SELECT есть хотя бы одна агрегатная функция (например, SUM), то все остальные колонки в SELECT обязаны быть перечислены в GROUP BY.

NULL в группах: GROUP BY считает все NULL одинаковыми и соберет их в одну отдельную группу «Неизвестно»

Инструмент №38: Создание групп из групп (группы внутри группы) 
GROUP BY col#1, col#2
Выполняется, если создание одной группы имен недостаточно, тогда мы добавляем в блок GROUP BY новые колонки, создавая тем самым комбинированные групп.
Пример:
GROUP BY city, month. Тогда система создаст уникальные пары (Москва-Январь, Москва-Февраль). Это позволяет детализировать отчет, не теряя мощности агрегации.

Создавая комбинированные группы, система выделяет каждой уникальной паре (Москва-Январь) свою строку. 
Закон: все колонки из GROUP BY обязаны дублироваться в SELECT, иначе отчет разобьется на невидимые группы и потеряет логику схлопывания. 
Если мы группируем по Городу и Месяцу, но в SELECT вывел только Город — ты увидишь несколько строк «Москва» с разными суммами и не поймешь, почему они не схлопнулись (а они разбиты по месяцам, которых не видно).
Порядок колонок в GROUP BY не влияет на математику СУБД, финальную структуру вложенности наводим через ORDER BY.



 
Глава 12. ФУНКЦИИ АГРЕГАЦИИ
MAX, MIN, AVG, SUM, COUNT

Функции агрегации (агрегаторы) – располагаются в блоке SELECT. Агрегаторы - это математические операции (найти максимальное, минимальное, среднее - значение, сложить или посчитать) которые выполняются в рамках столбца, если агрегация применилась без GROUP BY, выводимый результат будет получен в рамках всех значений заявленного столбца (при этом игнорируя пустоты); 
При этом, в случае агрегатора COUNT (подсчет количества) без заявленного столбца, значением для счета будет является не конкретная ячейка, а вся строка таблицы, соответственно COUNT (*) считает вообще все живые (существующие строки) таблицы, для него одна пустая ячейка ничего не значит., 
-  COUNT(*) — считает абсолютно все живые строки в таблице.
-  COUNT(col) — считает существующие значения в колонке, игнорируя пустоты (NULL).
Если в запросе фигурирует GROUP BY, то результат будет получен в рамках выполнения математических операций внутри созданных групп – из чего следует: если в SELECT есть хотя бы одна агрегатная функция (например, SUM), то все остальные колонки в SELECT обязаны быть перечислены в GROUP BY.
С GROUP BY агрегаторы сотрудничает с каждой отдельной группой, не перемешивая результаты. 

Важные нюансы:
– Нельзя выводить обычную колонку рядом с агрегатом без GROUP BY;
– Согласно очередности выполнения запроса (SELECT в последнюю очередь) – все функции агрегации выполняется также в последнюю очередь, а значит, что результат их выполнения будет получен на основе тех строк, что осталось после отсева.


 
Глава 12. ФУНКЦИИ АГРЕГАЦИИ
COUNT

Инструмент №39: Посчитать количество  строк – [COUNT (*)]
COUNT (*) – считает ВООБЩЕ ВСЕ строки таблицы. Указывая (*), система не привязывается к колонке;
Два режима работы:
– Без GROUP BY – считает вообще все строки;
– С GROUP BY считает количество (живых/существующих) строк (значений) для каждой созданной группы.
Маневр профи: Сравнивая COUNT(*) и COUNT(col). Если цифры разные — в таблице существуют «дыры» (NULL), так мы исключаем ручной перебора.


Инструмент №40: Посчитать количество значений – [COUNT (col)]
COUNT(cal) считает количество существующих значений в рамках указанной колонки.

Два режима работы:
– Без GROUP BY – считает вообще все существующие значения из указанной колонки;
– С GROUP BY – считает существующие значения в рамках каждой отдельной группы, заявленной колонки.

 
Глава 12. ФУНКЦИИ АГРЕГАЦИИ
COUNT (DISTINCT(COL))

Инструмент №41: Количество уникальных
[COUNT (DISTINCT)]
Применяется, если нужно узнать количество только уникальных «имен» в указанной колонке, 
Можно сказать: «игнорирует повторения», 
Если применили без GROUP BY - считает количество потенциальных групп в заявленной колонке.
Если применили с GROUP BY – считает уникальных, внутри групп.
COUNT (DISTINCT column) игнорирует NULL.

COUNT(DISTINCT) – это одна из самых тяжелых операций для процессора и оперативной памяти (RAM) сервера. На таблицах от миллиона строк он жрет много оперативной памяти, так как вынужден сортировать данные в RAM для поиска дублей. Если запрос начинает тормозить, этот маневр «переплавляют» через CTE-цех: сначала схлопывают таблицу через быстрый GROUP BY до уникальных значений, а в основном запросе делают легкий COUNT(*).
 
Глава 12. ФУНКЦИИ АГРЕГАЦИИ
SUM(COL)

Инструмент №42: Собрать сумму – [SUM (col)]
Применяя SUM, мы обязаны указать колонку, иначе агрегатор рухнет.

Агрегатор SUM работает только с числовым типом данных, помним про грязные данные и принудительную переплавку (числа, лежащие в виде текстового формата).

Если применили без group by – просуммирует все то, что располагается в указанной колонке, и выдаст результат в отдельной колонке (применяем алиас AS);
Если применили с group by – просуммирует данные в созданных группах, у каждой группы свой собственный результат, данная функция сотрудничает с каждой группой отдельно, не перемешивая данные между собой., 

Внутри SUM можно проводить математику до суммирования.
Пример: SUM (price * quantity).

Для предотвращения обнуления последующих расчетов, ВСЕГДА оборачиваем агрегатор SUM в COALESCE. 
 
Глава 12. ФУНКЦИИ АГРЕГАЦИИ
AVG(COL)

Инструмент №43:Найти среднее – [AVG]
AVG (col) – ищет среднее значение из указанной колонки: складывает все значения и делит на общее количество, при этом игнорирует пустоты (NULL).
Существует два сценария, которые зависят об бизнес задачи:
1) Если нам нужно учитывать «холостой» факт (например нечто, не оставившее числовой след, но оставившее факт существования  – «клиент зашел и но не купил», мы обязаны обернуть агрегатор COALESCE внутрь AVG, так, наше среднее охватит всех, даже «холостых».
Маневр профи:
Сравнивание AVG(col) и AVG(COALESCE(col, 0)). Если разница велика, значит, в системе много «пустышек», которые сильно завышают (или занижают) реальную картину эффективности.
2) Если нам, при нахождении среднего, из формулы нужно исключить «холостой факт», тогда COALESCE под запретом, просто применяем AVG, игнорируя холостой ход (пустоты).
*Холостой факт – факт существования без значения.

 
Глава 12. ФУНКЦИИ АГРЕГАЦИИ
MAX - MIN

Инструмент №44: Найти самое максимальное/минимальное [MAX/MIN]
MAX/MIN возвращает одно единственное значение из указанной колонки, либо самое MAX, либо самое MIN.
SELECT категория, MAX(цена) AS самое_большое
FROM table
GROUP BY категория
Без group by, покажет самое высокое значение из заявленной колонки,
С group by, покажет самое высокое значение внутри каждой группы отдельно., 

Типы данных: В отличие от SUM,  агрегаторы MAX и MIN работают с датами и текстом.
MAX для дат — это самая свежая дата;
MAX для текста — последнее слово по алфавиту;

Если нужна вся строка, в которой находится максимальное значение - используем ORDER BY цена DESC LIMIT 1.
 
Глава 13. СОРТИРОВКА:
ORDER BY

Инструмент №45: Сортировка, [ORDER BY]
	Сортировка выполняется в самом конце, когда все процессы в запросе выполнились, и нужно просто причесать результат., 
Сортировку можно выполнить в фоне, по колонке, не попавшей в итоговый результат, тогда тот порядок, в котором выстроятся строки в фоне, будет проецироваться на выводимый результат.
Синтаксис:
SELECT колонка_с_данными
FROM ТАБЛИЦА
ORDER BY колонка_с_данными DESC/ASC
ORDER BY идет после SELECT, можно сортировать по новым именам и вычислениям из SELECT.
NULL в сортировке: По умолчанию пустота (NULL) часто считаются самыми маленькими (в начале при ASC) или самыми большими (в начале при DESC). Если они мешают, используют надстройку NULLS LAST (в конец) или NULLS FIRST (в начало).
Синтаксис простой: надстройка ставится в самом конце блока ORDER BY, сразу после направления сортировки (ASC или DESC).
ORDER BY колонка [ASC/DESC] NULLS LAST -- Все пустые строки упадут в самый низ
ORDER BY колонка [ASC/DESC] NULLS FIRST -- Все пустые строки всплывут в самый верх

Можно сортировать по логическому условию. Например, мы можем выкинуть определенные строки в начало списка, написав условие: ORDER BY (цена > 1000) DESC. Система посчитает это выражение как TRUE (больше) и FALSE (меньше), подняв дорогие товары наверх. Однако - важный закон: чтобы внутри этой поднятой группы данные не превратились в хаос, всегда через запятую указываем второй приоритет: 
ORDER BY (цена > 1000) DESC, цена ASC. 
Тогда сначала всплывут дорогие товары, и внутри них они выстроятся по возрастанию цены.

Множественная сортировка: Можно сортировать по нескольким колонкам сразу: ORDER BY 1 DESC, 2 ASC. Сначала сработает первый приоритет, затем внутри него — второй. Важно – для каждого из множества сортировки требуется уточнение напраления (DESC/ASC).
 
Глава 14. ВЕСЫ ДЛЯ ГРУПП:
HAVING

Инструмент №46: Весовое сито групп – [HAVING]
HAVING – это фильтр, который работает строго со результатами, полученными внутри групп – при явном использовании GROUP BY.
Синтаксис:
SELECT city, SUM(sales) AS total_sales
FROM orders
GROUP BY city
HAVING SUM(sales) > 100000;

Смысл:
WHERE фильтрует сырые строки до группировки. HAVING фильтрует (взвешивает) уже готовые группы по результатам их агрегатов (SUM, COUNT, AVG) И ТОЛЬКО ИХ.
Внутри HAVING нужно писать агрегатную функцию целиком 
HAVING SUM(sales) > 10
Алиас из SELECT система там не увидит, так как HAVING выполняется раньше SELECT.
 
Глава 15. БЛОК SELECT:
 UNION - ВЕРТИКАЛЬНЫЙ МОСТ

Инструмент №47: Таблица над таблицей — [UNION / UNION ALL]
Оператор UNION располагается между двумя независимыми приказами SELECT и склеивает их результаты вертикально (добавляет строки вниз, увеличивая длину отчета).
Синтаксис:
SELECT material, quantity 
FROM supply_2024

UNION ALL

SELECT material, quantity 
FROM supply_2025

Существует два режима сшивки:
1. UNION ALL — Просто приклеивает нижние строки к верхним. Все дубликаты выживают.
2. UNION — Склеивает строки и принудительно удаляет полные дубликаты. 

Законы сборки:
1. Закон одинакового количества: Число колонок в верхнем и нижнем SELECT должно быть строго равным.
2. Закон однородных типов: Столбцы на одинаковых позициях должны совпадать по формату (число под числом, текст под текстом).
3. Диктатура имени: Заголовки колонок в итоговом отчете берутся строго из первого (верхнего) SELECT.
Если нужно упорядочить весь итоговый склеенный отчет (например, по дате или цене), блок ORDER BY пишется всего один раз в самом конце.

 
Глава 16. БЛОК SELECT: 
ТОЧКА СИНХРОНИЗАЦИИ NOW

Инструмент №48: Точка синхронизации, [NOW () в SELECT]
Функция NOW(), расположенная в SELECT, создает колонку и выводит одно единственное значение – дату и время актуального запроса. Нужно, если отчет нужно зафиксировать как контрольную точку проверки:
- Либо для сравнения значений из прошлых дат (взгляд назад)
- Либо для сравнения из будущего с настоящей контрольной точкой синхронизации (взгляд из будущего).

Функция NOW() располагается в блоке SELECT или WHERE и мгновенно возвращает текущую дату и точное время сервера на момент запуска запроса в формате YYYY-MM-DD HH:MM:SS (тип данных TIMESTAMP)
Синтаксис:
SELECT NOW() AS текущее_время;

*NOW() выдает время самого сервера СУБД

В PostgreSQL время NOW() фиксируется в момент старта запроса и не меняется до самого его окончания. Сколько бы минут ни выполнялся тяжелый расчет, для всех строк отчета NOW() вернет одинаковую секунду. 

Если нужен реальный тикающий таймер хода выполнения, используют функцию clock_timestamp()
 
Глава 17. БЛОК SELECT: 
ИЗВЛЕЧЕНИЕ ДЕТАЛЕЙ

Инструмент №49: Срез дат — [EXTRACT() в SELECT]
Функция EXTRACT располагается в блоке SELECT. Она заходит в ячейку с полной датой, забирает оттуда чистую изолированную цифру: только ГОД [YEAR], только МЕСЯЦ [MONTH] или только ДЕНЬ [DAY], и выводит её в отдельную колонку в виде обычного целого числа, не забываем про AS.
Синтаксис 
SELECT EXTRACT(что_вырезаем FROM колонка_с_датой)
1.	Вырезаем год: EXTRACT(YEAR FROM s_date) – вернет число 2026.
2.	Вырезаем месяц: EXTRACT(MONTH FROM s_date) – вернет число 5.
3.	Вырезаем день: EXTRACT(DAY FROM s_date) – вернет число 17.
Если требуется отфильтровать отчет по конкретному месяцу, мы обязаны писать функцию целиком: WHERE EXTRACT(MONTH FROM date_fact) = 5. Иначе будет ошибка 

Инструмент №50: Сжатие по периодам — [EXTRACT() в GROUP BY]
Самый мощный аналитический маневр. Когда мы вырезали из дат только номера месяцев, все дни превратились в одинаковую цифру месяца. Теперь GROUP BY видит эти урезанные месяца от дат как одинаковые строки и схлопывает их в один узел.
SELECT 
    EXTRACT(MONTH FROM date_fact) AS номер_месяца, 
    SUM(quantity) AS общий_объем
FROM supply
GROUP BY 1;

В случае существования в отчете нескольких лет, чтобы года не перемешались в процессе группировок, мы режем дату сразу по двум осям — и по Году, и по Месяцу одновременно:
SELECT 
    EXTRACT(YEAR FROM date_fact) AS год,
    EXTRACT(MONTH FROM date_fact) AS месяц, 
    SUM(quantity) AS объем
FROM supply
GROUP BY 1, 2; 
-- Сгруппирует по уникальным парам (2025-5, 2026-5)
Глава 18. БЛОК SELECT: 
ОБНУЛЕНИЕ ЧАСОВЫХ ПОЯСОВ

Инструмент №51: Обнуление времени — [DATE_TRUNC() в SELECT]
Главная роль DATE_TRUNC — это зануление времени (часов, минут, секунд) из единого формата TIMESTAMP.
По умолчанию базы данных в формате TIMESTAMP сохраняют дату вместе со временем (2026-05-17 14:32:05). Из-за разницы в секундах GROUP BY никогда не сможет объединить эти строки. 
DATE_TRUNC — это обнуление времени (топор дат). Указывая границу (например, день 'day'), система обнуляет всё, что находится правее этой границы, превращая часы и минуты в нули. На выходе остается полноценный календарный день.
Синтаксис:
DATE_TRUNC('до_какого_уровня_режем', колонка_с_датой)
На выходе DATE_TRUNC оставляет тип данных TIMESTAMP

Три режима физической обрезки (Наглядный пример):
Исходное грязное значение: 2026-05-17 14:32:05 (Воскресенье)
1.	Обрезка до Дня ('day'): DATE_TRUNC('day', s_date)
o	Результат: 2026-05-17 00:00:00. (Время выжжено в полночь. Идеально для ежедневных сводок).
2.	Обрезка до Месяца ('month'): DATE_TRUNC('month', s_date)
o	Результат: 2026-05-01 00:00:00. (Время выжжено, а день сброшен на первое число. Удобно для ежемесячных отчетов).
3.	Обрезка до Года ('year'): DATE_TRUNC('year', s_date)
o	Результат: 2026-01-01 00:00:00. (Остался только чистый старт года).

В реальном бизнесе (особенно на стройке или производстве) планирование и отчетность завязаны на НЕДЕЛИ - недельно-суточные графики, объемы за неделю. Отсюда вытекает необходимость в явной привязке к неделям 
4.	Обрезка до Недели ('week'): DATE_TRUNC('week', s_date) 
Результат: 2026-05-11 00:00:00. (Время выжжено, а дата автоматически сброшена на Понедельник текущей недели. Незаменимо для недельных графиков и поиска недельных трендов).

Разница в EXTRACT и DATE_TRUNC заключается в том, что первый «выкусывает» значение, а второе обнуляет.
EXTRACT — выкусывает кусок. Из полной даты 2026-05-17 он достает чистую изолированную цифру 5 (номер месяца). Прошлое и будущее стираются, остается только голая цифра для точечной фильтрации или сборки комбинированных осей.
DATE_TRUNC — обнуляет масштаб (топор дат). Из полной даты со временем он выжигает часы и минуты, сбрасывая масштаб до первого числа месяца: 2026-05-01 00:00:00. Календарная шкала сохраняется, позволяя строить сквозные графики динамики по периодам.
 
Глава 18. БЛОК SELECT: 
ГРУППИРОВКА ДЛЯ ОБНУЛЕНИЯ ЧАСОВЫХ ПОЯСОВ

Инструмент №52: Группировка для TIMESTAMP — [DATE_TRUNC() и GROUP BY]
Применяется, когда в базе лежат тысячи операций, зафиксированных с точностью до секунды. Чтобы собрать их в аккуратный посуточный отчет, мы сначала обнуляем время до уровня дня, а затем группируем.
SELECT 
    DATE_TRUNC('day', date_fact) AS чистый_день,
    SUM(quantity) AS объем_за_сутки
FROM supply
GROUP BY 1
Результат: Все операции, выполненные 17 мая в 10:00, в 14:32 и в 23:59, получат одинаковый маркер 2026-05-17 00:00:00 и схлопнутся в одну единственную строку за эти сутки.

ВАЖНО — функция DATE_TRUNC является «родной» для СУБД PostgreSQL. Если в будущем придется работать на серверах MySQL или SQL Server (MS SQL), там этого «топора» нет, а выжигание времени до уровня дня выполняется через другие инструменты (например, CAST(col AS DATE) в SQL Server или DATE(col) в MySQL). 

Всегда помни, на каком движке СУБД развернуто твое поле боя.!

 
Глава 19. БЛОК ОКОННЫХ ФУНКЦИ:
ОКОННЫЕ ФУНКЦИИ OVER (PARTITION BY ...) И ФУНКЦИИ АГРЕГАЦИИ

Любая оконная функция задается в блоке SELECT, формирует колонку под свой результат, которую мы обзываем при помощи AS, и по которой никогда нельзя фильтровать в блоке WHERE – для этого запрос с окнами сначала задаем в производной таблице (будь то CTE или запрос в блоке FROM), и только после, в главном запросе можем выполнять операции по фильтрации.

Инструмент №53: Виртуальный GROUP BY – [OVER (PARTITION BY)]
OVER (PARTITION BY) в связке с агрегатными функциями позволяет получить результат их выполнения физически не меняя таблицу (не схлопывая, как это делает GROUP BY), каждая строка и каждая деталь отчета остаются на своих местах. 

Оконные функции располагаются в блоке SELECT. Согласно очередности выполнения запроса, они вычисляются сервером ОДНИМИ ИЗ ПОСЛЕДНИХ — сразу после блоков GROUP BY/HAVING, но строго ДО финальной сборки SELECT, и итогового ORDER BY всего запроса. Из-за этого внутри окон нельзя использовать новые алиасы (имена) колонок, созданные в этом же SELECT.

Надстройка PARTITION BY внутри конструкции OVER — это приказ системе виртуально изолировать каждую доступную группу друг от друга и провести внутри каждой группы соответствующую операцию по агрегации, затем, вернуть результат агрегации каждому участнику каждой существующей группы – идеально, если в рамках групп, нужно сравнить точечны факт и сумму группы целиком наглядно. 

SUM(col_for_sum) OVER (PARTITION BY col_for_group) – заготовка для суммы
AVG(col_for_avg) OVER (PARTITION BY col_for_group) – заготовка для среднего
COUNT(col_for_count) OVER (PARTITION BY col_for_group) – заготовка для счета






Синтаксис:
SELECT 
    object_name,
    material_name,
    quantity, 
    SUM(quantity) OVER (PARTITION BY object_name) AS всего_по_объекту
FROM supply;

Применяя оконные функции, в выводимом отчете мы можем видеть:
Имя объекта, название материала, объем точечной поставки и весь объем данного материала, посчитанный для каждого объекта (пример не корректный, материал может быть разрозненный – м2, м3, пример существует для наглядности.
Физика работы:
1. Система берет итоговую таблицу и, встречая PARTITION BY, виртуально изолирует каждую доступную группу.
2. Оконный агрегат (например, SUM) заходит внутрь каждой виртуальной группы отдельно и собирает там соответствующий итог, полностью игнорируя то, что происходит в соседних виртуальных группах.
3. Полученный результат система приклеивает новой колонкой обратно к каждой существующей строке, которая находилась внутри такой группы.

Суть:
Мы получаем результат групповых расчетов (видим собранные агрегатами итоги по категориям, объектам или месяцам), но при этом сохраняем детальный, построчный вид отчета. Данные не перемешиваются между отсеками, а структура таблицы не разрушается. 
Глава 19. БЛОК ОКОННЫХ ФУНКЦИ:
ОБХОД ЗАПРЕТА ФИЛЬТРАЦИИ ПО ОКОННЫМ ФУНКЦИЯМ

Инструмент №54: Фильтр оконного результата — [Обход запрета в WHERE]

Система запрещает писать оконные функции в блоке WHERE. Приказ типа WHERE SUM(quantity) OVER(...) > 100 вызовет мгновенный отказ сервера.

Почему? Потому что WHERE выполняется раньше, чем оконные функции рождаются. Система физически еще не знает результатов окна, когда пытается отсеять строки.

Решение №1 (Маневр Матрешки): Мы формируем вложенный запрос внутри FROM, тем самым создавая чистую производную таблицу, а далее, в главном запросе, принимая новую – целую таблицу, где результат окна становится обычной  колонкой, по которой можно фильтровать
Решение № 2. Применим СTE-цех, Мы выносим запрос с оконной функцией в изолированный CTE-блок (WITH). Окно полностью вычисляется в памяти сервера, формируя чистую таблицу. В основном запросе мы спокойно фильтруем этот результат в блоке WHERE, как обычное плоское число.
 
Глава 19. БЛОК ОКОННЫХ ФУНКЦИ:
ВРЕМЕННАЯ ВИРТУАЛЬНАЯ ТАБЛИЦА – «CTE-ЦЕХА»

Инструмент №55: Изолированный цех данных — [WITH (CTE)]

CTE (Common Table Expression — обобщенное табличное выражение) — это инструмент, который позволяет создать одну или несколько временных, предварительно очищенных таблиц прямо в оперативной памяти сервера перед запуском основного запроса. 

Синтаксический скелет:
WITH cte_ceh_1 AS (
    -- Первый изолированный цех: чистим и готовим данные
    SELECT object_name, material_name, SUM(quantity) AS total_q
    FROM supply
    GROUP BY object_name, material_name
),
cte_ceh_2 AS (
    -- Второй изолированный цех: берем лимиты или сметы
    SELECT object_name, material_name, quantity_estimate
    FROM estimates
)
-- Главный конвейер запроса: просто и чисто соединяем результаты цехов
SELECT 
    t1.object_name,
    t1.material_name,
    t1.total_q,
    t2.quantity_estimate
FROM cte_ceh_1 AS t1
LEFT JOIN cte_ceh_2 AS t2 
    ON t1.object_name = t2.object_name 
    AND t1.material_name = t2.material_name;





Физика работы (Маневр «Параллельные цеха»):
1. Команда WITH пишется на самой первой строчке запроса. Система заходит в первый блок (cte_ceh_1), выполняет его в памяти СУБД и вешает на полученный результат короткий ярлык.
2. Если через запятую объявлен второй цех (cte_ceh_2), система делает то же самое. При этом второй цех может использовать данные из первого, как настоящую таблицу.
3. В самом конце включается главный конвейер (основной SELECT). Для него результаты работы цехов выглядят как абсолютно чистые, готовые физические таблицы в базе данных.
В чем выгода:
1. Идеальная читаемость. Код больше не выглядит как трехэтажная вложенная матрешка. Он развернут в плоскую, логичную цепочку шагов (сначала подготовили одно, потом второе, в конце сшили).
2. Безопасность и Highload-оптимизация. Тяжелые расчеты или «мировые константы» вычисляются в памяти ровно ОДИН РАЗ, а не запускаются скрытыми циклами для каждой строки, от чего база могла «задохнуться».
 
Глава 19. БЛОК ОКОННЫХ ФУНКЦИ:
НУМЕРАТОР - ROW_NUMBER() OVER()

Инструмент №56: Сквозной счетчик строк — ROW_NUMBER() OVER()

ROW_NUMBER() OVER(…) – это оконная функция, которая генерирует непрерывную последовательность целых чисел (1, 2, 3, 4...) для каждой строки в рамках заданного виртуального «окна». Даже если две строки абсолютно идентичны по своему содержимому, ROW_NUMBER() присвоит им разные номера на основе их физического порядка в выводе.
Синтаксис:
sql
ROW_NUMBER() OVER (
    PARTITION BY колонка_группировки 
    ORDER BY колонка_сортировки ASC/DESC
) AS имя_колонки

•	ROW_NUMBER() — Сам приказ: «Включить счетчик». Скобки всегда пустые.
•	OVER — Дверь в оконный режим.
•	PARTITION BY — Стена отсека. Делит всю таблицу на изолированные группы. Счетчик сбрасывается в 1 при переходе в новый отсек.
•	ORDER BY + DESC/ASC - внутри окна физически сортирует данные, но делает это внутри оперативной памяти сервера и строго в границах своего изолированного отсека (PARTITION BY), чтобы выставить правильный порядок для присвоения номера. На глобальный порядок строк в финальном отчете это не влияет — за него отвечает только внешний ORDER BY в самом конце запроса.
•	
Ловушка: (Амбивалентность сортировки). *двузначность
Если две строки идентичны, а порядок задается только по одному критерию (столбцу), например только по дате — система выдаст им номера 1 и 2 случайным образом. При каждом перезапуске запроса они могут меняться местами.
Защита: 
Задаем в ORDER BY уникальный дополнительный критерий для однозначности
Например:
ORDER BY delivery_date ASC, id ASC – мы задаем порядок по дате, и тут же по ID.
 
Глава 19. БЛОК ОКОННЫХ ФУНКЦИ:
РАНГИ - DENSE_RANK() OVER()

Инструмент №57: Строгое ранжирование — DENSE_RANK() OVER()

Логика: Это оконная функция, которая присваивает ранг (место в рейтинге: 1, 2, 3...) каждой строке внутри заданного «окна». Если две или более строки имеют абсолютно одинаковые значения в колонке сортировки (ORDER BY), система выдает им одинаковое место. При этом следующая за ними уникальная строка получит строго следующий по порядку номер, без перескоков. Слово DENSE переводится как «плотный» — это значит, что числовая последовательность рангов никогда не разорвется.

Синтаксис:
SELECT 
DENSE_RANK() OVER (
    PARTITION BY колонка_группировки 
    ORDER BY колонка_ранжирования ASC/DESC
) AS имя_колонки
FROM tab

•	DENSE_RANK() — Приказ: «Присвоить места по плотной шкале». Скобки всегда пустые.
•	PARTITION BY — Стена отсека. Делит всю таблицу на изолированные группы.
Счетчик сбрасывается в 1 при переходе к следующей группе объектов.
•	ORDER BY — Весы ранжирования. Колонка, по значению которой система определяет, кто достоин первого места, а кто — второго.

Внутренняя физика процесса (Разница между RANK и DENSE_RANK)
1.	ROW_NUMBER() выдаст: 1, 2, 3, 4. - Строгий и последовательный конвейер
2.	RANK() выдаст: 1, 1, 1, 4. – «Спортивный зачет» с дырой. Выдал трем лидерам первое место, но «сожрал» 2-е и 3-е места. Следующий получил сразу 4-е).
3.	DENSE_RANK() выдаст: 1, 1, 1, 2. – Плотный зачет. Выдал трем лидерам первое место, но сохранил непрерывность цепочки. 
 
Глава 19. БЛОК ОКОННЫХ ФУНКЦИ:
ШПИОН – ШАГ НАЗАД (LAG() OVER())

Инструмент №58: Взгляд назад— LAG() OVER()

Логика: возвращает значение из указанной колонки той строки, которая находится выше (раньше) текущей строки на заданный шаг внутри виртуального окна. Она позволяет сравнить текущую ячейку с её прошлым состоянием.

Синтаксис:
SELECT 
LAG(целевая_колонка, шаг_сдвига, заглушка_NULL) OVER (
    PARTITION BY колонка_группировки 
    ORDER BY колонка_сортировки ASC/DESC

FROM tab

•	LAG ( - Приказ: смотреть назад (вверх)
•	целевая_колонка — колонка, значения которой будем подсматривать взглядом назад
•	шаг_сдвига — на сколько строк назад отступить. Если не указать, по умолчанию равен 1 (предыдущая строка).
•	заглушка_NULL — Обязательный параметр: что подставить, если строки сзади нет (для самой первой строки в окне). Если не указать, система вернет пустоту NULL.

•	для LAG/LEAD отсутствие ORDER BY — это смерть логики. Система будет брать значения случайных строк «сверху» или «снизу», превращая динамику в мусор.
Поэтому, ORDER BY в OVER(...) для LAG/LEAD обязателен на 100%. Система должна четко понимать, какой именно выставлен порядок, с целью истинного результата (мы принудительно задаем очередь, в рамках которой будем смотреть назад – или вперед, без этой очереди будет хаос и смысл в логике инструмента обнулится)

Глава 20. ПРОДВИНУТЫЕ ФУНКЦИИ АГРЕГАЦИИ
ВЫБОРОЧНОЕ СУММИРОВАНИЕ И СОЗДАНИЕ МНОЖЕСТВА КОЛОНОК ДЛЯ СУММИРОВАНИЯ ПО УСЛОВИЯМ (PIVOT ТАБЛИЦА)

Инструмент №60: Выборочное (условное) суммирование

Конструкция SUM(CASE ... END) в SQL используется для выборочного (условного) суммирования. Она позволяет сложить значения из строк, которые соответствуют конкретному условию, игнорируя остальные, при этом конструкция позволяет создавать множество столбцов с такими вычислениями по условию (Разворот таблицы (Pivot))
По отдельности, операторы SUM и CASE работают стандартно: CASE проверяет строки по очереди, присваивая ярлыки по условиям, а SUM складывает результат. 
Но стоит их объединить, их логическая модель превращается в новый инструмент – сумме по условию (суммирует только то, что соответствует заданному условию).

Процесс работы конструкции:
1.	Сканирование: СУБД идет по таблице строка за строкой.
2.	Фильтрация (CASE): Для каждой строки соответствующего столбца проверяется условие. 
- Если оно верно (TRUE), возвращается указанное число или значение из колонка (соответственно трушное значение будет учитываться в операции SUM) 
- Если оно неверно (FALSE), возвращается 0 (или NULL) и не учувствует в операции SUM.
3.	Агрегация (SUM): Все полученные из CASE значения складываются в одну итоговую сумму.

Синтаксис
SELECT 
    SUM(CASE 
            WHEN условие THEN колонка_или_число
            ELSE 0 
        END) AS название_поля
FROM таблица
Важно: ELSE - всегда 0. Если его опустить, не подошедшие под условие строки вернут пустоты NULL. Суммирование с NULL не сломает расчет (SUM игнорирует NULL), но явный ELSE 0 делает код и безопаснее.
Для конструкции PIVOT таблицы, закладываем в запрос несколько конструкций через запятую:
    SELECT
SUM(CASE WHEN … THEN … ELSE 0 END) AS col_1,
 	SUM(CASE WHEN … THEN … ELSE 0 END) AS col_2,
FROM tab

Пример для понимания механизма:
Запрос:
SELECT 
    SUM(CASE 
            WHEN category = 'Electronics' THEN amount 
            ELSE 0 
        END) AS electronics_total
FROM orders;

Как база данных видит процесс изнутри:
•	Строка 1: Категория 'Electronics'? Да. Возвращаем amount -> 1000.
•	Строка 2: Категория 'Electronics'? Нет. Срабатывает ELSE -> 0.
•	Строка 3: Категория 'Electronics'? Да. Возвращаем amount -> 2000.
•	Строка 4: Категория 'Electronics'? Нет. Срабатывает ELSE -> 0.

Финальный шаг (SUM): 1000 + 0 + 2000 + 0 = 3000.
Результат: 3000.
 
Глава 20. ПРОДВИНУТЫЕ ФУНКЦИИ АГРЕГАЦИИ
ВЫБОРОЧНОЕ СУММИРОВАНИЕ И СОЗДАНИЕ МНОЖЕСТВА КОЛОНОК ДЛЯ СУММИРОВАНИЯ ПО УСЛОВИЯМ (PIVOT ТАБЛИЦА)

Частые ошибки 
•	Забытый END: CASE должен закрываться ключевым словом END. Без него возникнет синтаксическая ошибка.
•	Путаница с COUNT: Если написать COUNT(CASE WHEN условие THEN amount ELSE 0 END), то база данных посчитает все строки – для счета принудительно задаем после THEN единицу (1), так система соберет количество по критериям (инструмент 56)
•	Суммирование внутри CASE: Запись вида CASE WHEN ... THEN SUM(amount) END ошибочна. Сначала должна идти групповая функция SUM, а внутри нее — построчный фильтр CASE.

 
Глава 20. ПРОДВИНУТЫЕ ФУНКЦИИ АГРЕГАЦИИ
ВЫБОРОЧНОЕ СУММИРОВАНИЕ И СОЗДАНИЕ МНОЖЕСТВА КОЛОНОК ДЛЯ СУММИРОВАНИЯ ПО УСЛОВИЯМ (PIVOT ТАБЛИЦА)

Инструмент №61: Подсчет количества строк по условию (Аналог COUNT)

Вместо суммирования бизнес-показателей (каких-либо значений) пишем THEN 1. Каждая подошедшая строка прибавит к сумме единицу.

-- Считаем отдельно количество дорогих покупок
SELECT 
    SUM(CASE WHEN amount >= 1000 THEN 1 ELSE 0 END) AS premium_orders_count,
-- собираем количество самых дорогих
    FROM orders;
 
Глава 20. ПРОДВИНУТЫЕ ФУНКЦИИ АГРЕГАЦИИ
ВЫБОРОЧНОЕ СУММИРОВАНИЕ И СОЗДАНИЕ МНОЖЕСТВА КОЛОНОК ДЛЯ СУММИРОВАНИЯ ПО УСЛОВИЯМ (PIVOT ТАБЛИЦА)

Инструмент №62: Разворот таблицы (Pivot)
Это самый частый кейс. Вместо вывода категорий в разные строки через GROUP BY, мы выводим их в отдельные столбцы.

Когда используется этот маневр для создания PIVOT-таблицы (разворота строк в колонки), почти всегда он будет использоваться в связке с обычным GROUP BY (например, сгруппировать по Году, а в SUM(CASE) разложить 12 месяцев в 12 отдельных колонок).

-- Было: строки. Стало: таблица с колонками под каждую категорию
SELECT 
    SUM(CASE WHEN category = 'Electronics' THEN amount ELSE 0 END) AS electronics_sales,
    SUM(CASE WHEN category = 'Apparel' THEN amount ELSE 0 END) AS apparel_sales
FROM orders;
 
Глава 20.2. ОКОННЫЕ ФУНКЦИИ: 
НАКОПИТЕЛЬНЫЙ ИТОГ (ROWS / RANGE)

Маневр «Лифт сумм» — последовательное накопление.
Применяется, когда в отчете нужно увидеть непрерывную динамику нарастающего накопления в рамках заданных границ (с какой по какую строчку учитываем в данный миг).

Универсальной шаблон
SELECT
SUM(колонка для сбора суммы) OVER (
    ORDER BY колонка для сортировки
    [ROWS | RANGE] BETWEEN [НАЧАЛО] AND [КОНЕЦ]) AS НАРАСТАЮЩИЙ_ИТОГ
FROM TAB1
 
Глава 20.2. ОКОННЫЕ ФУНКЦИИ: 
НАКОПИТЕЛЬНЫЙ ИТОГ (ROWS / RANGE) – ЛОВУШКА ПУСТОТЫ

При расчете накопительного итога критически важно управлять поведением неопределенных значений (NULL). Если в колонке сортировки (например, даты доставки) ломается логика, ломается и весь расчет.
В чем суть «Ловушки пустоты»?
Представим, что из-за сбоя системы или грязных данных у нескольких строк в колонке delivery_date образовалась пустота (NULL).
Когда оконная функция начинает двигаться по строкам сверху вниз (как лифт, собирающий пассажиров по этажам), порядок сортировки решает всё:
•	Сортировка по умолчанию: В PostgreSQL при ORDER BY ASC система считает NULL самым «большим» значением и отправляет его в самый низ. В других СУБД (или при инверсивной сортировке DESC) NULL может признаться самым «маленьким» и всплыть на самый верх.

•	Катастрофа «Лифта» на верхних этажах: Если строки с NULL всплывают наверх, наш «лифт вычислений» застревает на первом же этаже. В SQL любое сложение с пустотой дает пустоту (100 + NULL = NULL). Лифт берет первый NULL, прибавляет к нему следующую строку и... снова получает NULL. Начинается каскадное заражение: весь накопительный итог для всего окна превращается в сплошную пустую черную дыру.

•	Катастрофа на нижних этажах: Если NULL падает в самый низ, он не ломает предыдущие строки, но искажает финальную аналитику. Строки без даты приплюсуются в самом конце, нарушая хронологию временного графика.

Как занулить зазор: Защита в монолите
Чтобы обезопасить расчеты, необходимо жестко и явно прописать правила парковки для пустых значений с помощью конструкции NULLS LAST (в конец) или NULLS FIRST (в начало).




Глава 20.2. ОКОННЫЕ ФУНКЦИИ: 
НАКОПИТЕЛЬНЫЙ ИТОГ (ROWS / RANGE) – ЛОВУШКА ПУСТОТЫ

-- Пример запроса с защитой от NULL
SELECT
    <колонка_группировки>,      -- Например: ID клиента, Название товара, Склад
    <колонка_сортировки>,      -- Например: Дата операции, Время, Номер чека
    <изменяемая_величина>,     -- Например: Сумма покупки, Количество, Вес
        -- Безопасный расчет накопительного итога
    SUM(<изменяемая_величина>) OVER (
        PARTITION BY <колонка_группировки> 
        ORDER BY <колонка_сортировки> ASC NULLS LAST 
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW 
    ) AS <название_накопительного_итога>

FROM <ваша_таблица_или_представление>; 
Глава 20.2. ОКОННЫЕ ФУНКЦИИ: 
ВЫБОР ДВИЖКА ЛИФТА: ROWS vs RANGE

Инструмент №63: ROWS (Построчный режим)
Суть: Считает строго физические строчки. Базе данных плевать на сами значения, она считает штуки (записи).
Эффект: Оценивает ФИЗИЧЕСКИЕ СТРОКИ таблицы. Режим ROWS полностью игнорирует данные в ячейках дат. Для него важен только физический номер строки. Лифт двигается строго построчно. Даже если вся база завалена дубликатами времени, нарастающий итог будет расти идеально плавно и последовательно: 
Если Строка 1 = 10, Строка 2 = 20, а Строка 3 = 30, то лифт посчитает так:
Строка 1: 10
Строка 2: 10 + 20 = 30
Строка 3: 30 + 30 = 60

Инструмент №64: RANGE (Групповой режим)
Суть: Считает логические значения. Ищет дубликаты в колонке ORDER BY и склеивает их в один блок.
Эффект: Сервер ориентируется на значения в колонке сортировки (на даты или секунды). Если несколько строк имеют одинаковую дату, режим RANGE объединит их в один неделимый куст. Сервер посчитает сумму всех этих строк за раз и запишет эту итоговую сумму во все совпавшие строки. Ход лифта станет дерганым, динамика внутри суток сотрется. 
Если Строка 1 = 10, Строка 2 = 20, а Строка 3 = 30 имеют одинаковое время, сервер мгновенно сложит их все вместе (10 + 20 + 30 = 60) и выдаст одно и то же число для каждой строчки:
Строка 1: 60
Строка 2: 60
Строка 3: 60
Конструкция с RANGE по умолчанию — это полный аналог простого ORDER BY без указания границ. Записи SUM(amount) OVER (ORDER BY date) и SUM(amount) OVER (ORDER BY date RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) абсолютно идентичны для сервера. Если опустить ключевое слово ROWS, база данных применит RANGE автоматически.
 
Глава 20.2. ОКОННЫЕ ФУНКЦИИ: 
ВАРИАНТЫ КОНСТРУКЦИЙ ДЛЯ ГРАНИЦ

Вариант конструкции для границ №1 – От самой первой строчки до текущего мига

BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW

Дешифровка операторов:
1.	BETWEEN ... AND ... — Задать рамки обзора от и до.
2.	UNBOUNDED PRECEDING — Начало отсека. Утверждает системе, что учитывать нужно самую первую физическую строчку группы или включить данные с самого старта таблицы.
3.	CURRENT ROW — Граница отсека, текущая строка. Ряд, на котором сервер стоит в данный миг. Означает: остановить расчет строго на текущей строке, не заглядывая вниз.

Вариант конструкции для границ №2 – Кастомизированный конструктор
Для настройки лифта под нестандартные задачи можно гибко менять параметры [НАЧАЛО] и [КОНЕЦ] внутри конструкции BETWEEN:
•	N PRECEDING — Отступить на N шагов назад от текущего мига. Используется для скользящих окон, например, сумма за 3 последних строки.
•	N FOLLOWING — Заглянуть на N шагов вперед в будущее. Захватить строки, которые сервер еще не прошел.

•	UNBOUNDED FOLLOWING — Строгая финальная точка: идти до самой последней строчки таблицы или секции. Используется, когда нужно сравнить текущую строку с финальным итогом.

Примеры быстрой сборки кастомных окон:
ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
Роль Скользящее окно: Текущий ряд + 2 ряда назад (всего 3 строки).
ROWS BETWEEN CURRENT ROW AND 5 FOLLOWING
Роль Взгляд в будущее: Текущий ряд + 5 рядов вперед.
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
Роль Полный обзор: От самого начала до самого конца. Сумма всей таблицы.
