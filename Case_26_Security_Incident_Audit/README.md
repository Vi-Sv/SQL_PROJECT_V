# Кейс №26: Коррелированные подзапросы и логический датчик ИБ-аудита

## Блок: Высшая Лига SQL / Информационная безопасность
**Заказчик:** Руководитель департамента ИБ / Аудитор внутренней безопасности холдинга.

### Бизнес-задача (ТЗ):
> *"Для проведения плановой аттестации и формирования реестра благонадежного персонала необходимо составить верифицированный список сотрудников подразделения 'Аналитика', не имеющих зафиксированных критических нарушений регламентов информационной безопасности. Требуется нормализовать ФИО сотрудников (удалить пробелы, привести к единому стандарту). Из итоговой выборки необходимо полностью исключить персонал, имеющий в логах инцидентов записи с уровнем угрозы 'CRITICAL'. Скрипт должен быть оптимизирован под Highload-нагрузки. Результат отсортировать по алфавиту."*

---

## Исходная структура СУБД (Таблицы и ключи сопряжения):

### 1. Штатное расписание (`staff_accounts`):
*   `emp_id` (INT) — Уникальный номер сотрудника (Первичный ключ).
*   `emp_name` (VARCHAR) — ФИО сотрудника (содержит грязный ручной ввод).
*   `department` (VARCHAR) — Структурное подразделение / департамент.

### 2. Лог инцидентов безопасности (`security_incidents`):
*   `incident_id` (INT) — Уникальный номер факта нарушения (Первичный ключ).
*   `emp_id` (INT) — Идентификатор сотрудника (Внешний ключ сопряжения со справочником `staff_accounts`).
*   `severity_level` (VARCHAR) — Уровень критичности инцидента (например, 'LOW', 'MEDIUM', 'CRITICAL').

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического конвейера была применена модель бесконтактного прерывания вычислений через коррелированный подзапрос `NOT EXISTS`. Данный паттерн кардинально превосходит классические операторы исключения множеств (`NOT IN`, `EXCEPT`) на Highload-объемах данных, поскольку не создает промежуточных списков в ОЗУ и полностью застрахован от падений при встрече с системными пустотами `NULL`.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Инициализация базового контура (`WITH clean_staff`):** Сервер запускает сканирование справочника сотрудников `staff_accounts` (`t1`), фильтруя строки по признаку `department = 'Аналитика'`. Скалярный инструмент `TRIM(LOWER())` стандартизирует ФИО аналитиков, удаляя пробельный шум.
2. **Включение логического датчика (`NOT EXISTS`):** Для каждой отфильтрованной строки аналитика движок СУБД запускает быстрый подзапрос к таблице нарушений `security_incidents` (`t2`). Конструкция `SELECT 1` минимизирует объем переносимых данных. Жесткий коррелированный мост сопряжения внутренних и внешних строк (`t2.emp_id = t1.emp_id`) проверяет наличие флага `severity_level = 'CRITICAL'`.
3. **Оценка триггера и фильтрация:** Если критическая запись в логе нарушений обнаружена — внутренний датчик мгновенно останавливает сканирование строки и возвращает значение `TRUE`, а оператор `NOT EXISTS` полностью аннигилирует (стирает) сотрудника из контура. Если нарушений нет — датчик возвращает `FALSE`, и сотрудник заносится в чистый белый список (в Python Pandas логика реализована через инвертированный векторный метод `~df.isin()`).
4. **Вывод стерильной витрины:** Финальный внешний запрос обращается к результатам CTE-цеха и упорядочивает белый список по алфавиту посредством конструкции `ORDER BY ASC`.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и верификации ИБ-конвейера данных были зафиксированы и устранены следующие технические риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за удвоения операторов):** Падение компилятора базы данных на этапе парсинга из-за дублирования ключевых слов `WHERE` и `AND` (`AND WHERE NOT EXIST`, `AND AND t2.severity_level`), вызванного утомлением глаз и набором кода на клавиатуре смартфона вслепую. Дополнительно зафиксирован пропуск буквы `S` в слове `EXISTS`.
    *   *Метод устранения:* Лишние дубли операторов моторики полностью удалены из черновика, синтаксический контур приведен к строгому промышленному канону.
*   **Риск №2 (Ловушка пропущенной закрывающей скобки цеха CTE):** Ошибка компиляции на стыке слоев из-за отсутствия внешней круглой скобки, завершающей работу всего обобщенного выражения `clean_staff`. Подзапрос датчика `NOT EXISTS` был закрыт корректно, но внешняя магистраль осталась разомкнутой.
    *   *Метод устранения:* Контур скобок верифицирован Ключом №3, парная закрывающая скобка жестко интегрирована перед финишным оператором `SELECT`.
*   **Риск №3 (Косметическая опечатка символьного маркера витрины):** Пропуск буквы `и` в наименовании финишного алиаса столбца вывода отчета (`"Стерильные_аналитки"`).
    *   *Метод устранения:* Буквенный зазор моторики устранен, имя колонки скорректировано до канонического вида `"Стерильные_аналитики"`.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная стерильная витрина ИБ-аудита интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного контроля кадровой безопасности холдинга.

### Структура дашборда:
1. **Белый список департамента аналитики (Verified Staff List):** Интерактивная таблица, выводящая ФИО сотрудников, успешно прошедших сквозную проверку логическим датчиком ИБ, отсортированная по алфавиту, служащая базой для начисления квартальных премий.
2. **Виджет уровня кадровой чистоты подразделения (KPI Card):** Процентный информер, вычисляющий долю «стерильных» сотрудников от общей штатной численности выбранного отдела компании.
3. **Модуль контроля критических рисков:** Фильтр-панель для оперативного переключения масок уязвимостей и выгрузки списков нарушителей безопасности для департамента комплаенса.
