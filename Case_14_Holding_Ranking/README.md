# Кейс №14: Оконное ранжирование ROW_NUMBER и вырезание топ-3 крупных платежей

## Блок: Аналитика холдингов / Коммерческий аудит
**Заказчик:** Генеральный директор.

### Бизнес-задача (ТЗ):
> *"Для оценки коммерческой эффективности региональной сети холдинга и презентации результатов инвесторам необходимо сформировать аналитический срез крупнейших транзакций. Требуется вывести нормализованные наименования регионов, идентификаторы клиентов и чистые суммы их платежей. По каждому отдельному региону необходимо изолировать строго ТОП-3 самых крупных платежей, полностью исключив из финального реестра все последующие операции. Итоговую витрину отсортировать по алфавитному порядку регионов, а внутри них — по возрастанию рейтинга от 1 до 3."*

---

## 🛠 Исходная структура СУБД (Таблица `regional_payments`):
*   `payment_id` (INT) — Уникальный номер чека / транзакции (Первичный ключ).
*   `region_name` (VARCHAR) — Название регионального филиала (содержит грязный ручной ввод регистров и пробелов).
*   `client_id` (INT) — Уникальный идентификатор контрагента.
*   `pay_amount` (INT) — Сумма внесенного платежа в рублях (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе технического анализа сырого массива данных `regional_payments` были локализованы три ключевые уязвимости:
1. **Текстовая неконсистентность филиалов:** Наличие скрытых пробелов и хаотичного регистра символов разорвет расчет аналитических функций на ложные группы;
2. **Системные пустоты финансовых метрик:** Пропуски `NULL` в суммах платежей искажают ранжирование и блокируют математический движок;
3. **Ограничение агрегации в ANSI SQL:** Запрет на использование оконных функций внутри стандартного фильтра `WHERE` верхнего уровня.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Запуск подзапроса (`FROM`):** Движок СУБД обращается к физической таблице `regional_payments`, поднимает сырой массив транзакций в кэш оперативной памяти и выполняет предварительную нормализацию комбо-резцами `TRIM(LOWER())` с занулением `NULL` через `COALESCE`.
2. **Секционирование географических зон:** Команда `PARTITION BY` на лету нарезает таблицу на изолированные, герметичные сектора по чистым названиям регионов для независимого расчета.
3. **Запуск рангового автомата:** Внутри каждого регионального окна оператор `ORDER BY DESC` выстраивает накладные по убыванию сумм. Ювелирный нумератор `ROW_NUMBER()` принудительно штампует сквозные порядковые ранги 1, 2, 3... от максимальных чеков к минимальным. Результат запечатывается во внутреннем поле под техническим именем `g4` подзапроса `TAB2` (в Python Pandas данная логика реализована через встроенный метод `.cumcount() + 1`).
4. **Хирургическое сито `WHERE` (КПП верхнего уровня):** Подзапрос закрывается скобкой и выкатывается наверх. Фильтр главного верхнего этажа `WHERE TAB2.g4 IN (1, 2, 3)` мгновенно, как лазером, срезает и выбрасывает в утиль абсолютно все строки, чей ранг упал ниже 3-го места в каждом городе.
5. **Вывод стерильной витрины:** Финальный контур `ORDER BY 1 ASC, 4 ASC` структурирует вывод по алфавиту регионов, внутри которых ранги последовательно распределяются от 1 до 3.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и верификации двухэтажного рангового конвейера были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за капкана незакрытой скобки):** Падение компилятора базы на этапе парсинга из-за пропущенной парной закрывающей круглой скобки после аргумента `LOWER(region_name)` непосредственно перед оператором сортировки внутри оконного пространства.
    *   *Метод устранения:* Структура скобок восстановлена, синтаксический контур стабилизирован.
*   **Риск №2 (Рыночный эталон — текстовые кавычки алиасов витрины):** Сбой парсинга при ошибочном использовании одинарных текстовых кавычек `'` для переименования колонок на верхнем уровне `SELECT` (`AS 'ID_клиента'`). В спецификации SQL одинарные кавычки зарезервированы строго под текстовые константы, а алиасы требуют использования двойных кавычек `""`.
    *   *Метод устранения:* Одинарные текстовые маркеры заменены на промышленные двойные кавычки.
*   **Риск №3 (Коллизия номеров осей в итоговом ORDER BY):** Логический перекос вывода вследствие ошибочной прописи сортировки по номерам столбцов `ORDER BY 1, 3 DESC`. Код упорядочивал суммы денег по убыванию, но ломал визуальный контроль рейтинга 1, 2, 3 для инвесторов.
    *   *Метод устранения:* Номера управляющих осей переориентированы на целевой ранг: `ORDER BY 1 ASC, 4 ASC`.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная ранговая витрина холдинга интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для презентации результатов инвесторам.

### Структура дашборда:
1. **Топ-3 транзакций по регионам (Leaderboard Chart):** Многоуровневая столбчатая диаграмма (Группированный Bar Chart), наглядно выводящая тройку крупнейших чеков в разрезе каждого географического филиала.
2. **Карта концентрации крупного капитала:** Сравнительный график долей, отражающий, какой процент от общего финансового потока региона занимают топ-3 зафиксированных платежа.
3. **Панель экспресс-фильтрации регионов:** Интерактивный селектор для мгновенной изоляции конкретного города и детального изучения карточек клиентов, совершивших максимальные транзакции.
