# Кейс №13: Расчет накопительного итога (Running Total) и блокировка кассовых разрывов

## Блок: Аналитика ритейла / Финансовый мониторинг
**Заказчик:** Коммерческий директор.

### Бизнес-задача (ТЗ):
> *"Для оперативного выявления и предотвращения кассовых разрывов необходимо спроектировать инструмент поминутного мониторинга входящего финансового потока. Требуется сформировать реестр продаж с выводом точных дат операций и чистой величины выручки (пропуски занулить). В третьем смежном столбце необходимо реализовать расчет накопительного итога (суммы с нарастающим итогом по хронологии времени) для непрерывного отслеживания объема кассы предприятия без сжатия детальных строк чеков. Вывод упорядочить от старых транзакций к новым."*

---

## Исходная структура СУБД (Таблица `shop_sales`):
*   `sale_id` (INT) — Уникальный номер чека (Первичный ключ).
*   `sale_date` (TIMESTAMP) — Точная дата и время совершения торговой операции.
*   `revenue` (INT) — Сумма выручки в рублях (содержит системные пропуски `NULL`).

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе предварительного технического аудита сырого массива данных `shop_sales` были локализованы три критические уязвимости:
1. **Риск декомпозиции математики из-за NULL:** Пропуски в ячейках выручки блокируют корректную работу оконных агрегаторов;
2. **Баг «схлопывания дат» (Групповые скачки):** При наличии транзакций с идентичной временной меткой стандартная оконная сортировка суммирует их оптом в одну ступень, полностью искажая поминутную кумулятивную динамику кассы.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Изоляция среды (`FROM`):** Движок СУБД обращается к физической таблице `shop_sales`, разворачивая сырой массив строк в оперативной памяти сервера для последующей конвейерной обработки.
2. **Зануление атомарных пустот (`COALESCE`):** Функция `COALESCE(revenue, 0)` перехватывает каждую пустую ячейку `NULL` на этапе сканирования колонок и преобразует её в цифровой ноль (0), подготавливая консистентную базу для суммирования.
3. **Развертывание глобального окна (`OVER`):** Ввиду умышленного отсутствия конструкции `PARTITION BY`, СУБД отключает секционирование и создает единое сквозное аналитическое пространство на весь массив данных.
4. **Хронологическое упорядочивание и пошаговый расчет:** Конструкция `ORDER BY sale_date, sale_id` внутри окна выстраивает транзакции в строгую временную цепочку. Включение уникального `sale_id` вторым ключом сортировки заставляет движок воспринимать каждую строку как уникальную единицу времени. Это активирует правило `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, вычисляя сумму ювелирно (Строка 1 -> 1+2 -> 1+2+3) и полностью блокируя ступенчатые скачки на совпадающих датах.
5. **Формирование витрины (`SELECT`):** Агрегатор `SUM` фиксирует результаты в поле `running_total_revenue`. Строки не сжимаются — детальный отчет сохраняет 100% первоначального лога транзакций ритейла. Глобальный финальный оператор `ORDER BY sale_date ASC` выстраивает итоговую таблицу в хронологическом порядке.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования кумулятивного конвейера данных были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Критический сбой ядра — групповое схлопывание дат):** Логический сбой расчета нарастающего итога из-за прописи оконной сортировки только по одному полю `ORDER BY sale_date`. При совпадении времени транзакций СУБД выдавала ложный резкий скачок суммы вместо плавного пошагового прироста.
    *   *Метод устранения:* В оконный контур сортировки интегрирован уникальный ключ `sale_id` в качестве жесткого стабилизатора шага вычислений.
*   **Риск №2 (Потеря предохранителя в блоке вывода SELECT):** Локальная ошибка моторики, при которой пропуски `NULL` были занулены внутри оконного агрегатора `SUM(COALESCE(...))`, но на верхний уровень вывода была вызвана голая сырая колонка `revenue`. Накопительный итог считался верно, но во втором столбце отчета отображались системные пустоты.
    *   *Метод устранения:* Конструкция `COALESCE` продублирована для автономного поля вывода в первой секции `SELECT`.
*   **Риск №3 (Синтаксическая ловушка кроссплатформенных алиасов):** Попытка вызова финальной сортировки по имени расчетного алиаса `ORDER BY "Динамика" ASC`. Ряд легаси-движков СУБД (Oracle, старые версии MSSQL) обрушивают компиляцию, так как не умеют читать алиасы `SELECT` в итоговом `ORDER BY`.
    *   *Метод устранения:* Для универсальности и отказоустойчивости кода в кроссплатформенных средах финишная сортировка привязана к физическому полю таблицы. В СУБД PostgreSQL исходный вариант стабилен по умолчанию [91.18].

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование финансового дашборда)
Очищенная кумулятивная витрина ритейла интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для непрерывного контроля ликвидности.

### Структура и состав дашборда:
1. **График нарастающего итога кассы (Running Total Area Chart):** Накопительная диаграмма с областями, отражающая поминутный или подневный прирост живых денежных средств компании, позволяющая руководству визуально контролировать динамику капитала.
2. **Индикатор суточной выручки (KPI Card):** Крупный цифровой виджет, отображающий чистый операционный доход за текущий день (`SUM(clean_revenue)`) для экспресс-оценки коммерческой активности.
3. **Таблица транзакционных пиков:** Таблица, сортированная по убыванию индивидуальных чеков, для быстрого выявления аномально крупных покупок, резко меняющих угол наклона графика кумулятивной суммы.
