# Кейс №21: Вложенные CTE-магистрали и каскадная фильтрация

## Блок: Высшая Лига SQL / Финансовый аудит
**Заказчик:** Коммерческий директор / Руководитель финансового департамента.

### Бизнес-задача (ТЗ):
> *"Для оптимизации накладных расходов холдинга и выявления аномально высоких арендных платежей необходимо спроектировать многоступенчатый изолированный аналитический конвейер. Требуется провести предварительную нормализацию данных из лога офисных затрат (очистить регистры и пробелы, занулить системные пропуски). На этапе промежуточной обработки необходимо изолировать транзакции, относящиеся исключительно к статье расходов 'аренда'. На финальную витрину вывести реестр городов и сумм, где чистая величина платежа строго превысила лимит в 150 000 единиц, упорядочив результат по алфавиту."*

---

## Исходная структура СУБД (Таблица `office_expenses`):
*   `expense_id` (INT) — Уникальный номер расходной операции (Первичный ключ).
*   `office_city` (VARCHAR) — Наименование географического филиала (содержит грязный ручной ввод).
*   `expense_item` (VARCHAR) — Статья накладных расходов (содержит регистровый шум).
*   `cost` (INT) — Фактическая сумма затрат в рублях (содержит пропуски `NULL`).

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического скрипта была развернута многоуровневая Enterprise-модель — последовательные вложенные обобщенные табличные выражения. Декларация `WITH t1 AS (...), t2 AS (...)` позволяет разбить сложный бизнес-процесс на изолированные технологические цеха, снижая нагрузку на вычислительное ядро СУБД и исключая избыточные повторные сканирования жесткого диска.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Развертывание базовой магистрали (`WITH pure_data`):** Сервер СУБД инициализирует первое изолированное пространство в ОЗУ. Комбо-резцы `TRIM(LOWER())` стандартизируют текстовые поля городов и статей расходов, приводя их к единому регистру, а оператор `COALESCE` трансформирует пропуски `NULL` в чистые нули, предохраняя математический контур от падения.
2. **Запуск каскадного цеха (`WITH filtered_cities`):** Через разделительную запятую движок открывает вторую зависимую магистраль. Этот цех обращается напрямую к очищенным результатам предыдущего шага `FROM pure_data` (в Python Pandas логика последовательно реализована через передачу датафрейма `pure_df` на вход `filtered_df`), полностью блокируя избыточные обращения к исходному тяжелому диску. На этом этапе применяется фильтрационное сито `WHERE g3 = 'аренда'`, полностью вычищающее маркетинг, канцелярию и прочие побочные статьи расходов.
3. **Выход на финишный этаг (`FROM filtered_cities`):** Главный внешний запрос вызывает стерильный массив второго цеха. Фильтр верхнего этажа `WHERE filtered_cities.f2 > 150000` отсекает фоновый операционный шум, пропуская на витрину только аномально высокий финансовый жир аренды.
4. **Сортировка вывода:** Готовая структурированная таблица выстраивается по алфавитному порядку городов посредством оператора `ORDER BY ASC`.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования многоступенчатого каскадного конвейера были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за зазора моторики):** Падение парсера компиляции базы данных на самом старте запроса из-за механического пропуска буквы `C` в ключевом управляющем операторе `SELECT` (`SELET`), вызванного высокой скоростью ввода в блокноте.
    *   *Метод устранения:* Синтаксис восстановлен, имя команды приведено к валидному промышленному стандарту `SELECT`.
*   **Риск №2 (Логический разрыв каскадной цепи CTE):** Тяжелый зазор утомления глаз, при котором во втором цехе `filtered_cities` был прописан ошибочный повторный вызов исходной грязной таблицы `FROM office_expenses` вместо передачи очищенного потока `FROM pure_data`. Это приводило к падению сервера из-за попытки вызова полей `g1, g2, g3`, физически отсутствующих в сырой СУБД.
    *   *Метод устранения:* Проведен сквозной аудит логики, контур каскада жестко замкнут на первый очищенный цех `FROM pure_data`.
*   **Риск №3 (Логический капкан нестрогого порога ТЗ):** Риск несанкционированного включения в отчет сумм, равных ровно 150 000 единиц, при жестком требовании заказчика вывести строго превысившие лимит операции.
    *   *Метод устранения:* В финальном блоке фильтрации применен строгий знак превосходства (`>`), обеспечивший абсолютную точность выводимой коммерческой метрики.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная каскадная витрина финансового аудита под управлением CTE интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для непрерывного контроля бюджетов.

### Структура дашборда:
1. **Витрина критических затрат на аренду (KPI Leaderboard):** Интерактивная таблица «Город -> Сумма аренды», автоматически аккумулирующая филиалы, преодолевшие установленный финансовый фильтр, отсортированная по алфавиту, для мгновенной локализации пиковых расходов ФОТ компании.
2. **Сравнительный маркер региональной нагрузки (Bar Chart):** Столбчатая диаграмма, визуализирующая объемы арендных платежей в разрезе городов для экспресс-оценки коммерческим директором стоимости содержания локальных офисов.
3. **Виджет совокупного перерасхода (KPI Card):** Крупный цифровой информер, отображающий общую сумму зафиксированных аномальных превышений лимита по аренде для оперативного принятия решений по оптимизации договоров.
