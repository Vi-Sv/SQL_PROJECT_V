## 1. ТЕХНИЧЕСКИЙ СЛОВАРЬ (КАРТА СМЫСЛОВ)

* B-Tree (Balanced Tree / Сбалансированное дерево) — базовая структура хранения индекса, похожая на перевернутое дерево ветвления.
* СУБД (Система управления базами данных) — программа, которая управляет памятью и выполняет SQL-запросы (в нашем контексте — PostgreSQL).
* Движок / Планировщик (Query Planner) — внутренний мозг СУБД, который рассчитывает самый дешевый по времени и RAM путь выполнения запроса.
* Чтение с диска (I/O — Input/Output) — самая медленная физическая операция, когда сервер считывает блоки данных с жесткого диска в оперативную память.
* RAM (Random Access Memory) — сверхбыстрая оперативная память сервера.
* Страница (Page / Block) — минимальная неделимая единица обмена данными между диском и RAM в СУБД. В PostgreSQL 1 страница жестко равна 8 Килобайтам. Даже если тебе нужна одна строчка весом 100 байт, сервер физически прочитает с диска всю страницу в 8 Кб.
* Таблица Фактов (Fact Table) — гигантская таблица, куда валятся миллионы записей-действий (логи, транзакции, ежедневные объемы выполнения).
* Кардинальность (Cardinality) — степень уникальности данных в колонке (отношение уникальных значений к общему числу строк).
* Селективность (Selectivity) — доля строк, которую возвращает фильтр WHERE от общего объема таблицы. Выражается от 0 до 1.

## 2. ОПУСТОШЕНИЕ КОРОБКИ: ИЗОЛЯЦИЯ ЭЛЕМЕНТОВ
Границы коробки: Рассматриваем физику хранения, структуру B-Tree, механику поиска и сценарии, когда СУБД блокирует индекс. Не лезем в специфические индексы для геопозиций (GIST) и полнотекстового поиска (GIN). Только то, что нужно аналитику на терабайтах данных.
## Элемент №1: Физическая природа индекса
Индекс — это отдельный служебный файл на диске, который жестко привязан к конкретной таблице.
Таблица на диске лежит как хаотичный склад коробок (Страниц по 8 Кб). Строки туда дописываются в конец по мере поступления.
Индекс — это алфавитный указатель. Он дублирует в себя только две вещи из таблицы: значение индексируемой колонки и TID (Tuple ID / физический адрес строки на диске) в формате (номер страницы, номер строки на ней).
## Элемент №2: Внутренний механизм B-Tree (Как устроено сбалансированное дерево)
Если в таблице 10 000 000 строк, индекс не хранит их плоским списком (иначе его самого пришлось бы долго читать). Он строит иерархию уровней:

                  [ Корневой узел: Числа от 1 до 1000 ]
                                 /      \
       [ Промежуточный: 1-500 ]          [ Промежуточный: 501-1000 ]
               /          \                         /          \
  [Лист: 1-250]      [Лист: 251-500]    [Лист: 501-750]    [Лист: 751-1000]


* Корневой узел (Root Node): Вершина. Сюда заходит поисковый робот СУБД. Содержит ссылки на промежуточные узлы.
* Промежуточные узлы (Internal Nodes): Маршрутизаторы. Они говорят роботу: «Искомое число больше 500? Тебе направо».
* Листовые узлы (Leaf Nodes): Самый нижний уровень. Только здесь хранятся реальные значения колонок и указатели на физические адреса страниц на диске (TID).

Физика поиска: Движку нужно найти строку с id = 720. Вместо перебора 10 миллионов строк, он заходит в Корень -> прыгает в промежуточный узел 501-1000 -> прыгает в Лист 510-750. За 3 шага в памяти он нашел адрес строки на диске, пошел и забрал ее.
## Элемент №3: Виды сканирования в планах EXPLAIN
Когда аналитик проверяет тяжелый запрос через команду EXPLAIN SELECT..., планировщик показывает, как именно он будет шуршать по диску:

   1. Seq Scan (Последовательное сканирование): Индекс отсутствует или проигнорирован. Сервер тупо поднимает с диска в RAM абсолютно все 8-килобайтные страницы таблицы подряд.
   2. Index Scan (Сканирование по индексу): Робот идет по дереву B-Tree, находит нужные Листы, забирает адреса строк, а затем идет в файл таблицы и точечно выдергивает эти строки с диска.
   3. Index Only Scan (Сканирование только по индексу): Самый быстрый маневр. Если твой SELECT запрашивает только те колонки, которые уже входят в состав индекса, робот берет данные прямо из Листов B-Tree. Ему вообще не нужно ходить в основной файл таблицы на диске.
   4. Bitmap Index Scan + Bitmap Heap Scan (Гибрид): Применяется, когда надо вытащить много строк (например, 15% таблицы). Робот сначала идет по индексу, собирает адреса всех нужных строк в оперативной памяти, сортирует эти адреса по номеру физических страниц на диске (чтобы читать диск строго последовательно, без хаотичных прыжков головки жесткого диска), а потом одним плавным проходом (Bitmap Heap Scan) собирает данные.

## 3. СТРАТЕГИЯ АНАЛИТИКА: ОШИБКИ И ПРАВИЛА УСКОРЕНИЯ## Смертельный капкан: Почему СУБД игнорирует индекс (И уходит в Seq Scan)
Ты создал индекс по колонке status_date. Но СУБД в упор его не видит в 5 случаях:

   1. Низкая селективность (Теория 10%): Если твой фильтр WHERE запрашивает данные, которые занимают больше 10–15% от общего объема таблицы, планировщик скажет: «Прыгать по дереву индекса туда-сюда за адресами миллионов строк дороже, чем один раз прочитать всю таблицу целиком через Seq Scan». Индекс отключается сам.
   2. Использование функций на индексируемой колонке:
   * Код-убийца: WHERE DATE_TRUNC('month', status_date) = '2026-07-01' — индекс умер. Движок не может применить дерево к результату функции.
      * Код-Middle: WHERE status_date >= '2026-07-01' AND status_date < '2026-08-01' — индекс работает идеально.
   3. Неявное приведение типов (Маскировка):
   * Если колонка object_code имеет тип VARCHAR (строка), а ты пишешь: WHERE object_code = 1044 (передал число), СУБД под капотом начнет преобразовывать каждую строчку таблицы к числу. Индекс блокируется. Пиши строго в типе колонки: WHERE object_code = '1044'.
   4. Поиск по маске с процентом в начале (LIKE):
   * WHERE material_name LIKE 'Труба%' — индекс работает (робот знает, с каких букв начинается поиск в дереве).
      * WHERE material_name LIKE '%108мм' — индекс умер (робот не знает начальную точку в B-Tree, приходится перебирать всё).
   5. Ловушка OR:
   * WHERE id = 500 OR номенклатура_id = 12 — если по номенклатура_id индекса нет, СУБД проигнорирует даже индекс по id и уйдет в полный перебор всей таблицы. Заменяй такие конструкции на объединение запросов через UNION.
   
## Проектирование: Составные индексы (Composite Indexes)
Если ты часто фильтруешь данные по двум колонкам одновременно (например, WHERE объект_id = 5 AND тип_работ = 'Сварка'), обычные одиночные индексы работают неэффективно. Нужен один составной индекс на две колонки: CREATE INDEX idx_name ON tab (объект_id, тип_работ);.
Жесткое правило порядка колонок в составном индексе:
Колонки внутри индекса должны идти в порядке убывания их уникальности (от высокой кардинальности к низкой), либо строго соответствовать логике твоих запросов слева направо.

* Если у тебя индекс построен на (объект_id, тип_работ), то запрос WHERE объект_id = 5 задействует этот индекс.
* Но запрос WHERE тип_работ = 'Сварка' (без указания объекта) этот индекс использовать не сможет, потому что дерево B-Tree разветвлено в первую очередь по объектам. Ты не можешь перепрыгнуть первый уровень ветвления.

## 4. РЕЗЮМЕ ДЛЯ АНАЛИТИКА
Аналитик не создает индексы каждую минуту (это работа DBA / Дата-инженеров). Задача аналитика — не ломать существующие индексы своими запросами.
Когда твой тяжелый расчет по Таблице Фактов вешает базу:

   1. Запусти EXPLAIN перед запросом.
   2. Ищи в тексте плана надпись Seq Scan на больших таблицах.
   3. Смотри в секцию Filter: если там видишь преобразования типов, функции (LOWER, TRUNC) или дикие условия OR — переписывай код так, чтобы колонка оставалась «голой» и чистой.
