# Кейс №12: Вертикальный анализ LEAD и проектирование графика зазоров снабжения

## Блок: Промышленная аналитика / Логистика и Снабжение ПТО
**Заказчик:** Начальник отдела снабжения ПТО.

### Бизнес-задача (ТЗ):
> *"Необходимо разработать детализированный аналитический график ритмичности поставок сырьевых строительных материалов (цемент, щебень) для предотвращения простоев на объектах. Требуется вывести унифицированные наименования материалов, дату текущей отгрузки и фактический объем в тоннах. В смежном столбце необходимо рассчитать точную временную метку следующей по времени поставки этого же типа материала для визуализации логистического интервала (зазора) между рейсами. Строки отчета не должны подвергаться группировке или сжатию. Вывод упорядочить по номенклатуре и возрастанию дат."*

---

## 🛠 Исходная структура СУБД (Таблица `materials_delivery`):
*   `delivery_id` (INT) — Уникальный номер транспортной накладной (Первичный ключ).
*   `material_name` (VARCHAR) — Наименование строительного материала (содержит грязный ручной ввод регистров и пробелов).
*   `delivery_date` (TIMESTAMP) — Точная дата и время фактического прибытия груза на объект.
*   `tonnage` (INT) — Объем поставки в тоннах.

---

## Дескриптор логики решения (Технический аудит):

В ходе предварительного анализа структуры данных `materials_delivery` были локализованы три ключевые системные уязвимости:
1. **Риск неконсистентности номенклатурных групп:** Хаотичный ручной ввод регистра букв операторами на весовой приведет к ложному дроблению однородных материалов при расчете оконных функций;
2. **Риск декомпозиции временной шкалы:** Отсутствие строгого хронологического порядка внутри расчетных блоков приведет к хаотичным зачетам дат из случайных периодов;
3. **Строгая типизация полей (Конфликт типов данных):** Попытка принудительной подстановки числового нуля (`0`) в качестве дефолтного значения для финальных строк выборки вызовет аппаратный сбой компилятора базы данных из-за несовместимости типов `TIMESTAMP` и `INT`.

### Алгоритм построения изящного конвейера:
1. **Предварительная текстовая нормализация:** С помощью комбинации `TRIM(LOWER(material_name))` наименования материалов приводятся к единому стандарту, ликвидируя скрытые пробелы и регистровые ошибки.
2. **Секционирование и упорядочивание оконных зон:** Конструкция `PARTITION BY TRIM(LOWER(material_name))` изолирует временные ряды поставок каждого материала в независимые расчетные перегородки. Внутри окон оператор `ORDER BY delivery_date ASC` выстраивает накладные по строгой хронологической шкале от старых к новым.
3. **Развертывание шпиона смещения вперед:** К потоку дат применяется аналитическая функция `LEAD(delivery_date, 1)`. Данный инструмент заглядывает на одну строчку вперед в будущее, извлекая дату следующего планируемого рейса без разрушения детальной структуры сметного тоннажа (в Python Pandas логика реализована через зеркальный инвертированный сдвиг вектора `.shift(-1)`). Дефолтное значение для финальных строк умышленно оставлено пустым (`NULL` / `NaT`), что гарантирует аппаратную стабильность типов данных СУБД.
4. **Финальное выравнивание:** Витрина структурируется общим оператором `ORDER BY` по возрастанию наименований и дат отгрузок.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования логистического конвейера были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за зазора моторики):** Падение компилятора базы на этапе парсинга из-за пропущенного разделительного знака запятой (`,`) в конце четвертой строки после объявления алиаса `tonnage AS "Объем"`, непосредственно перед вызовом оконного оператора `LEAD`.
    *   *Метод устранения:* Проведен синтаксический аудит, запятая-разделитель интегрирована в структуру селекта.
*   **Риск №2 (Логический перекос витрины отчета):** Ошибочное написание глобальной финальной сортировки `ORDER BY delivery_date DESC` по убыванию дат. Код отрабатывал корректно, однако витрина выворачивалась задом наперед, ломая логистам визуальный контроль интервалов шага вперед.
    *   *Метод устранения:* Финальная сортировка принудительно перестроена на возрастание (`ASC`), восстановив естественный ход времени в графике снабжения.
*   **Риск №3 (Критический конфликт типов данных — Блокировка компиляции):** Попытка зашивки числового нуля в качестве дефолтного аргумента для граничных значений временного ряда: `LEAD(delivery_date, 1, 0)`. СУБД выдавала фатальную ошибку несовместимости типов данных, блокируя выполнение всего Highload-сервера.
    *   *Метод устранения:* Числовой ноль был удален из параметров функции. Для финальных строк конвейера оставлен легитимный пустой `NULL`, что обеспечило полную аппаратную стабильность контура.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная логистическая витрина ПТО интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для контроля ритмичности поставок на объекты.

### Структура дашборда:
1. **График зазоров снабжения (Time Gap Analyzer):** Линейно-столбчатая диаграмма, вычисляющая и отображающая разницу в днях/часах между текущей и последующей датой поставки (`next_delivery_date - current_delivery_date`). Позволяет начальнику снабжения визуально фиксировать критические разрывы в цепочке.
2. **Сводный информер ритмичности поставок:** Сводная интерактивная таблица, группирующая средний тоннаж отгрузок по дням недели для оптимизации работы складских терминалов.
3. **Модуль экстренной фильтрации номенклатуры:** Панель с быстрыми кнопками («Цемент», «Щебень», «Арматура») для мгновенного вывода детальной хронологической ведомости по критическому материалу на экран.
