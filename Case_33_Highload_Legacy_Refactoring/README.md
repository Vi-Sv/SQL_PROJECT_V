# Кейс №33: Рефакторинг легаси-кода через CTE-магистрали и плоский Pivot-агрегат

## Блок: Высшая Лига SQL / Рефакторинг и Оптимизация СУБД
**Заказчик:** Директор по технологиям (CTO) / Коммерческий директор холдинга.

### Бизнес-задача (ТЗ):
> *"Существующий легаси-запрос аналитической отчетности вызывает критическую перегрузку оперативной памяти сервера и зависание базы данных вследствие выполнения тяжелых скалярных подзапросов в блоке SELECT для каждой строки. Необходимо провести радикальный рефакторинг архитектуры скрипта с применением Бритвы Оккама. Требуется сформировать оптимизированную витрину поквартального оборота активных клиентов. На финальный экран вывести имена пользователей со статусом 'active', а также обособленные столбцы чистых сумм их покупок по направлениям 'IT' и 'AUTO' (пропуски занулить). Вычисления выполнить за один физический проход вычислительного контура по таблице логов. Результат отсортировать по алфавиту."*

---

## Исходная структура СУБД (Две таблицы и ключи сопряжения):

### 1. Реестр пользователей (`users`):
*   `users_id` (INT) — Уникальный номер клиента (Первичный ключ).
*   `user_name` (VARCHAR) — Имя / ФИО пользователя.
*   `status` (VARCHAR) — Текущий статус аккаунта в системе холдинга (например, 'active', 'banned').

### 2. Лог заказов (`orders`):
*   `id_orders` (INT) — Уникальный номер чека (Первичный ключ, гарантирующий уникальность каждой операции).
*   `users_id` (INT) — Идентификатор пользователя (Внешний ключ сопряжения со справочником `users`; содержит многократные дубли продаж).
*   `cat` (VARCHAR) — Товарная категория совершенной покупки (например, 'IT', 'AUTO', 'FOOD').
*   `amount` (INT) — Сумма транзакции в рублях (содержит системные пропуски `NULL`).

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе устранения критического архитектурного бага легаси-кода была развернута ультимативная модель Высшей Лиги SQL — предварительное условное агрегирование (Conditional Pivot Aggregation) внутри обобщенного табличного выражения (CTE) с последующим левосторонним сопряжением. Это ликвидирует циклическое сканирование жесткого диска СУБД, заменяя миллионы избыточных чтений одним последовательным проходом вычислительного ядра по индексам.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Инициализация изолированного цеха (`WITH select_sum`):** До старта основного селекта движок СУБД разворачивает виртуальное пространство в оперативной памяти (ОЗУ). База заказов `orders` сканируется один раз. Оператор `GROUP BY users_id` принудительно сжимает многотысячный транзакционный лог до уникальных ключей пользователей.
2. **Внедрение Pivot-переключателей и двойной рубеж защиты:** Внутри группировки на лету разворачиваются два плоских условных сумматора `SUM(CASE WHEN)`. Если зачищенная скалярными комбо-резцами `TRIM(LOWER())` категория совпадает с маской нижнего регистра ('it' или 'auto'), чек суммируется, иначе прибавляется ноль. Двойной рубеж защиты данных (`Data Quality`) — связка внутреннего и внешнего `COALESCE(..., 0)` — намертво страхует математику сервера от падений и гарантирует вывод стерильного числового нуля на витрину даже при полном отсутствии транзакций по контрагенту.
3. **Развертывание левостороннего моста сопряжения:** Главный внешний запрос обращается к физическому справочнику `users` (`t1`), фильтруя строки по критерию `t1.status = 'active'`. К очищенному пулу клиентов пришивается сжатая CTE-матрица сумм (`t2`) посредством оператора `LEFT JOIN` по ключу `users_id`. Стыковка проходит ювелирно «один-к-одному» без риска логического вспучивания данных, а левый мост гарантирует стопроцентное сохранение в отчете активных пользователей с нулевой историей покупок (в Python Pandas логика зеркально реализована через условный фильтр `np.where` и метод объединения матриц `pd.merge(..., how='left')`).
4. **Вывод витрины:** Финальный реестр выстраивается по алфавитному порядку имен пользователей с помощью `ORDER BY ASC`.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и финальной верификации юбилейного конвейера оптимизации были зафиксированы и устранены следующие технические риски и зазоры моторики:

*   **Риск №1 (Синтаксическая отсечка из-за капкана избыточной запятой перед FROM):** Падение компилятора базы данных на этапе парсинга подзапроса из-за символа запятой (`,`), ошибочно оставленного в конце списка полей первого цеха непосредственно перед ключевым словом `FROM orders`.
    *   *Метод устранения:* Лишний разделительный знак удален из структуры, синтаксический контур CTE стабилизирован.
*   **Риск №2 (Ловушка коллизии адресации чужих алиасов в главном SELECT):** Критическая ошибка парсинга на верхнем уровне запроса, при которой в блоке вывода была прописана попытка извлечь столбцы через ложное имя несуществующей таблицы `t3.g3` (`t2.g2 AS "IT", t3.g3 AS "AUTO"`), в то время как весь агрегированный цех подзапроса был задекларирован под единым алиасом `t2`.
    *   *Метод устранения:* Проведен точечный аудит адресов, вызов обеих расчетных Pivot-колонок переведен на строгий родительский префикс временного цеха: `t2.g3 AS "AUTO"`.
*   **Риск №3 (Капкан неоднозначности ключей и полей в блоках ON и WHERE):** Ошибка сопряжения таблиц из-за вызова исходного поля `t2.users_id` в блоке `ON`, переименованного внутри CTE в алиас `g1`. Дополнительно зафиксирован вызов голого неэкранированного префикса `users.status` в блоке фильтрации активных клиентов.
    *   *Метод устранения:* Цепь сопряжения моста жестко замкнута на внутреннее поле `ON t1.users_id = t2.g1`, а финальное сито переориентировано на верифицированный префикс справочника `t1.status = 'active'`.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование коммерческого дашборда)
Очищенная кросс-витрина рефакторинга СУБД интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного мониторинга эффективности клиентских сегментов.

### Структура дашборда:
1. **Сводная ведомость активности клиентов (Active Users Pivot Board):** Интерактивная многоколоночная таблица «Имена пользователей -> Выручка IT -> Выручка AUTO», выводящая клиентов со статусом 'active', прошедших сквозную очистку, со встроенным фильтром по нулевым значениям для выявления неактивного сегмента.
2. **Индикатор оптимизации серверной нагрузки (Performance Card):** Крупный цифровой виджет, отображающий кратную скорость прироста производительности выполнения отчетов (сокращение избыточных дисковых чтений I/O).
3. **Модуль быстрой фильтрации по именам:** Интерактивное текстовое поле ввода для мгновенной изоляции конкретного пользователя холдинга и проверки структуры его коммерческого оборота на экране.
