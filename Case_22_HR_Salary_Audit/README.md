# Кейс №22: СТЕ-агрегация и сквозной аналитический мост

## Блок: HR-аналитика / Финтех-Кадры
**Заказчик:** Руководитель финансового департамента / Директор по персоналу.

### Бизнес-задача (ТЗ):
> *"Для оптимизации фонда оплаты труда (ФОТ), проведения сравнительного анализа региональных ставок и выявления ключевых высокооплачиваемых специалистов подразделений необходимо разработать комплексный аналитический инструмент. Требуется сформировать ведомость с выводом персональных идентификаторов сотрудников, нормализованных названий департаментов и чистой величины их текущих окладов (пропуски занулить). В отдельном столбце необходимо вывести средний оклад их структурного подразделения. На итоговую витрину должны попасть исключительно те сотрудники, чей индивидуальный оклад строго превышает среднее значение по их департаменту. Вывод отсортировать по убыванию окладов."*

---

## Исходная структура СУБД (Таблица `employee_salaries`):
*   `emp_id` (INT) — Уникальный номер сотрудника (Первичный ключ).
*   `department_name` (VARCHAR) — Наименование структурного подразделения / департамента (содержит грязный ввод).
*   `salary` (INT) — Величина индивидуального оклада сотрудника (содержит системные пропуски `NULL`).

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического скрипта была развернута отказоустойчивая двухэтажная модель: предварительное агрегирование данных внутри обобщенного табличного выражения (CTE) с последующим внутренним сопряжением (`INNER JOIN`) по нормализованному ключу. Данный подход гарантирует высокую скорость обработки данных на Highload-нагрузках корпоративного уровня за счет минимизации объема сканирования строк на верхних этажах логики.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Инициализация вычислительного цеха (`WITH dept_averages`):** До выполнения основного селекта движок СУБД разворачивает изолированное виртуальное пространство в RAM. Скалярные комбо-резцы `TRIM(LOWER())` нормализуют названия департаментов, а `COALESCE` предохраняет расчеты от падения из-за пустот. Операторы `AVG` и `GROUP BY` сжимают массив данных до уникальных кодов подразделений с расчетом средних окладов, округляемых функцией `ROUND(..., 2)`.
2. **Развертывание моста данных «один-ко-многим» (`INNER JOIN`):** Исходная детальная таблица логов `employee_salaries` (`f1`) сопрягается со сжатой CTE-матрицей средних показателей (`f2`) через внутренний оператор склейки по нормализованному текстовому ключу департамента (в Python Pandas логика зеркально реализована через метод `pd.merge(..., how='inner')`). К каждой строчке сотрудника притягивается среднее значение его отдела.
3. **Хирургическое сито пост-фильтрации (`WHERE`):** Контур `WHERE COALESCE(f1.salary, 0) > f2.g2` выполняет функцию контрольно-пропускного пункта верхнего уровня. Он отсекает сотрудников с окладами ниже или равными среднему значению по цеху, пропуская на витрину только высокооплачиваемый топ-состав персонала.
4. **Сортировка витрины:** Финальный оператор `ORDER BY DESC` ранжирует итоговый VIP-реестр по убыванию величины чистой выручки / оклада сотрудников.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования каскадного конвейера ФОТ были зафиксированы и устранены следующие технические риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка округления `ROUND::NUMERIC`):** Падение парсера базы данных на этапе компиляции подзапроса из-за некорректного распределения скобок и разделительных запятых при попытке совместить встроенное округление и явное приведение типов (`ROUND(AVG(...))::numeric,2`). Параметры точности ошибочно оказались вне юрисдикции функции.
    *   *Метод устранения:* Контур скобок перепроектирован, приведение типов и шаг округления корректно упакованы внутрь аргументов: `ROUND(AVG(...)`::NUMERIC, 2).
*   **Риск №2 (Ловушка вызова функций через алиас таблицы перед именем оператора):** Критическая синтаксическая ошибка, вызванная хаотичной прописью префикса таблицы перед системными командами сервера СУБД (`f1.TRIM(LOWER(...))`). Движок принудительно обрушал выполнение, пытаясь обнаружить внутреннюю функцию внутри физической структуры таблицы.
    *   *Метод устранения:* Проведен точечный аудит адресации, префиксы таблиц перенесены внутрь скобок непосредственно к физическим полям базы: `TRIM(LOWER(f1.department_name))`.
*   **Риск №3 (Капкан неоднозначности полей в блоках WHERE и ORDER BY):** Логический сбой адресации на верхнем уровне селекта при попытке вызвать финальный текстовый алиас вывода через точку префикса таблицы (`ORDER BY f1."Оклад"`). Движок СУБД блокировал сортировку, поскольку на данном этапе оперирует исходными полями источника, а не финишными именами витрины.
    *   *Метод устранения:* Финальная сортировка привязана к чистому расчетному алиасу вывода `"Оклад"` без указания родительских таблиц, что полностью зацементировало контур в Postgres.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная витрина анализа ФОТ интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для кадрового контроля расходов.

### Структура дашборда:
1. **Интерактивный реестр высокооплачиваемых специалистов:** Детализированная таблица «ID сотрудника -> Департамент -> Личный оклад -> Средний оклад по цеху» со встроенным условным форматированием. Автоматически подсвечивает строки сотрудников, чье отклонение от средней нормы превышает 50%.
2. **Сравнительный график перерасхода ФОТ по подразделениям:** Столбчатая диаграмма (Bar Chart), отображающая совокупный объем превышения бюджетов в разрезе департаментов для принятия стратегических управленческих решений кадровым директором.
3. **Модуль быстрой селекции департаментов:** Панель с выпадающим списком уникальных наименований подразделений для мгновенной изоляции конкретного отдела холдинга и изучения структуры окладов на экране.
