# Кейс №28: Логическое ветвление CASE WHEN внутри агрегации

## Блок: Финтех-Куб / Коммерческая аналитика
**Заказчик:** Коммерческий директор.

### Бизнес-задача (ТЗ):
> *"Для комплексного анализа выполнения планов продаж и оценки продуктовой специализации персонала необходимо разработать многомерную Pivot-витрину. Требуется рассчитать совокупный макро-оборот по каждому менеджеру на основе лога транзакций (пропуски занулить). Одновременно с этим в смежных столбцах необходимо вывести чистые суммы продаж по целевым категориям 'электроника' и 'одежда'. Вычисления выполнить в один проход по таблице без каскадного сшивания миров. Из отчета полностью исключить отладочные записи тестовых аккаунтов. Результаты упорядочить по убыванию общего оборота."*

---

## Исходная структура СУБД (Таблица `store_transactions`):
*   `tx_id` (INT) — Уникальный номер торговой транзакции (Первичный ключ).
*   `manager_name` (VARCHAR) — ФИО коммерческого менеджера (содержит грязный ручной ввод).
*   `item_category` (VARCHAR) — Товарная категория проданной позиции.
*   `amount` (INT) — Сумма чека в рублях (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического скрипта был применен высокопроизводительный паттерн Enterprise-разработки — условная агрегация (Conditional Aggregation). Имплантация логических ветвлений `CASE WHEN` внутрь стандартных функций `SUM` позволяет реализовать динамическое перераспределение метрик по целевым колонкам за один физический проход сервера СУБД по таблице, полностью блокируя избыточную нагрузку на RAM и дисковую подсистему.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Первичная фильтрация (`WHERE`):** Движок СУБД выполняет сквозное сканирование таблицы `store_transactions` и на самом входе в систему отсекает технический и отладочный мусор по строковой маске `NOT LIKE '%тест%'`, снижая объем обрабатываемых данных.
2. **Кластеризация потока (`GROUP BY`):** Строки группируются по уникальным кластерам предварительно зачищенных комбо-резцами `TRIM(LOWER())` имен менеджеров.
3. **Параллельный запуск условных счетчиков:** Внутри каждого сформированного кластера менеджера движок запускает три параллельных суммирующих сумматора `SUM`. Базовый сумматор считает макро-оборот `SUM(COALESCE(amount, 0))`. Условные операторы `CASE` работают как стрелочные переключатели на лету: если нормализованная категория строго совпадает с маской нижнего регистра ('электроника' или 'одежда'), чек приплюсовывается в целевую ячейку, иначе прибавляется нейтральный ноль (0) (в Python Pandas логика реализована через векторный фильтр `np.where` и последующую агрегацию `.agg()`).
4. **Вывод Pivot-витрины:** Готовая многомерная таблица упорядочивается по строгому убыванию общего объема продаж посредством оператора `ORDER BY DESC`.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и верификации конвейера условной агрегации были зафиксированы и устранены следующие технические риски и ошибки сбоя:

*   **Риск №1 (Логический капкан регистра маски — тотальное зануление отчета):** Критическая логическая коллизия черновика, при которой строковые предикаты внутри оператора `CASE WHEN` были прописаны с заглавной буквы (`'Электроника'`, `'Одежда'`), в то время как к физическому полю на входе применялся жесткий скалярный очиститель `LOWER()`. Из-за регистрозависимости строк в СУБД условия сравнения никогда бы не совпали, и сервер вывел бы на витрину директора ложные нулевые результаты по всем категориям.
    *   *Метод устранения:* Текстовые маски проверки принудительно выровнены в строгий нижний регистр (`'электроника'`, `'одежда'`) в соответствии с правилами Data Quality.
*   **Риск №2 (Ловушка неэкранированных кириллических алиасов):** Риск падения кодировок компилятора при вызове длинных расчетных колонок кириллицей без экранирования в ANSI SQL (`AS Сумма_продаж_электроники`).
    *   *Метод устранения:* Все выходные русскоязычные наименования столбцов упакованы в жесткие промышленные двойные кавычки `""`.
*   **Риск №3 (Механический зазор моторики в имени столбца):** Пропуск буквы `н` в финишном названии макро-колонки отчета (`"Общая_сумма_всех_продаж_мееджера"`), снижающий презентабельность документа.
    *   *Метод устранения:* Буквенный пропуск устранен, алиас приведен к каноническому виду `"Общая_сумма_всех_продаж_менеджера"`.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная многомерная Pivot-витрина интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного аудита структуры продаж.

### Структура дашборда:
1. **Сводная матрица эффективности продаж (Sales Pivot Matrix):** Интерактивная многоколоночная таблица «Менеджер -> Общие продажи -> Продажи электроники -> Продажи одежды» со встроенным цветовым ранжированием (Heatmap) для экспресс-выявления ключевых сотрудников.
2. **Гистограмма продуктовой специализации (Grouped Bar Chart):** Группированная столбчатая диаграмма, сопоставляющая долевой вклад электроники и одежды в общей структуре выручки каждого менеджера.
3. **Виджет операционной зачистки:** Селектор для быстрой фильтрации лидеров по общему обороту холдинга.
