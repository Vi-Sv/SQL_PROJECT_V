# Кейс №2: Двухэтажная матрешка и логистическое сжатие дублей

## Блок: Логистика и управление цепями поставок
**Заказчик:** Руководитель отдела логистики.

### Бизнес-задача (ТЗ):
> *"Вследствие программного сбоя весового терминала накладные на отгрузку задублировались в системных логах с полной идентичностью данных. Это привело к искусственному раздуванию объемов поставок. Необходимо устранить последствия сбоя, полностью очистить транзитный реестр от дубликатов накладных и рассчитать корректный средний объем поставок в разрезе каждого города."*

---

## Исходная структура СУБД (Таблица `raw_shipments`):
*   `shipment_id` (INT) — Номер накладной доставки.
*   `city` (VARCHAR) — Город поставки (содержит хаотичный ручной ввод регистров и пробелов).
*   `provider_name` (VARCHAR) — Имя поставщика.
*   `ship_time` (TIMESTAMP) — Точное время операции секунда в секунду.
*   `volume` (INT) — Объем поставки в тоннах (содержит пропуски `NULL`).

---

## Дескриптор авторского ключа (Логика решения):

В ходе технического аудита сырой таблицы `raw_shipments` были выявлены три критические уязвимости данных:
1. Наличие полных стопроцентных дубликатов строк, совпавших секунда в секунду;
2. Ггрязный ручной ввод регистров и концевых пробелов в текстовых полях городов и имен поставщиков;
3. Наличие системных пустот и дыр `NULL` в объемах поставок.

Для решения задачи спроектирована двухэтажная аналитическая архитектура. 

Сначала формируется внутренний слой данных (виртуальная таблица `NEW_TAB` во `FROM`). Внутри этой «Матрешки» через оператор `GROUP BY` по всем колонкам одновременно схлопываются полные дубликаты строк, совпавшие секунда в секунду (в Python Pandas этот шаг изящно решен одной командой `.drop_duplicates()`). 

Параллельно внутри подзапроса инструментом `TRIM(LOWER)` срезается текстовый мусор пробелов и регистров в названиях городов и поставщиков, приводя их к единому стандарту, а пропуски в объемах (`volume`) локализуются и заменяются нулями через `COALESCE` для защиты конвейера вычислений от падения.

На втором (внешнем) этаже логики из очищенной виртуальной таблицы вызывается поле `CLEAN_CITY`, а через агрегатную функцию `AVG` вычисляется средний объем поставки с округлением результата до сотых долей при помощи `::NUMERIC(10,2)`. Финишный результат группируется по чистым географическим узлам.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная логистическая витрина подключается к BI-платформе (Apache Superset / Yandex DataLens) для мониторинга эффективности цепей поставок.

### Состав и структура дашборда:
1. **Сводная гео-таблица показателей:** Интерактивный реестр «Город поставки -> Средний объем поставки», позволяющий руководству мгновенно видеть чистую среднюю загрузку по ключевым региональным направлениям.
2. **Аналитическая карта поставок (Heatmap):** Карта с цветовым кодированием интенсивности отгрузок, где плотность цвета привязана к среднему тоннажу. Позволяет визуально оценить самые загруженные транспортные узлы.
3. **Фильтр-модуль «Поставщик» (Slicer):** Выпадающий список уникальных имен поставщиков для моментальной фильтрации дашборда и оценки средних объемов отгрузок конкретного контрагента по городам.
