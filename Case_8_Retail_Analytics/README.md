# Кейс №8: Сортировочный чит-код NULLS LAST и защита каталога в ритейле

## Блок: Аналитика ритейла / Коммерческий учет
**Заказчик:** Коммерческий директор.

### Бизнес-задача (ТЗ):
> *"Необходимо сформировать консолидированный аналитический отчет по всему каталогу номенклатуры товаров на основе данных из лога продаж. Требуется рассчитать общее количество проданных единиц по каждому наименованию продукции. Исключить риск искажения (вспучивания) итоговых сумм. Позиции из каталога, не имевшие продаж в отчетном периоде, должны быть сохранены в реестре со значением ноль (0) вместо системных пропусков. Итоговую витрину отсортировать по убыванию объемов продаж."*

---

## Исходная структура СУБД (Сопряжение таблиц один-ко-многим):

### 1. Каталог товаров (`products`):
*   `item_id` (INT) — Уникальный идентификатор товара (Первичный ключ).
*   `item_name` (VARCHAR) — Название товара (ручной хаотичный ввод данных).
*   `base_price` (INT) — Базовая стоимость единицы товара.

### 2. Лог продаж (`sales_logs`):
*   `sale_id` (INT) — Уникальный номер факта продажи (Первичный ключ).
*   `item_id` (INT) — Идентификатор товара (Внешний ключ связи; содержит многократные дублирующие записи по одной позиции).
*   `quantity` (INT) — Количество проданных штук в одном чеке (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Технический аудит):

В ходе проектирования архитектуры конвейера данных были локализованы три ключевые системные уязвимости:
1. **Риск логического вспучивания данных:** Прямое сопряжение справочника `products` с грязным транзакционным логом `sales_logs` через `LEFT JOIN` приводит к искусственному раздуванию строк и умножению итоговых показателей `SUM`, что срывает аудит коммерческой сети.
2. **Текстовая неконсистентность:** Риск размытия позиций при группировке из-за скрытых пробелов в названиях товаров.
3. **Образование пустых связей:** Появление системных пустот `NULL` на верхнем уровне селекта для позиций, которые ни разу не продавались в отчетном периоде.

### Алгоритм устранения хаоса данных:
1. **Предварительная агрегация (Внутренний слой подзапроса):** До этапа сопряжения таблиц лог продаж `sales_logs` изолируется во внутреннем подзапросе (`t2`) и принудительно сжимается через оператор `GROUP BY item_id` с подсчетом суммы `SUM(COALESCE(quantity, 0))`. К моменту объединения транзакционный массив сжат до уникальных ключей, что полностью устраняет риск дублирования строк на верхнем уровне.
2. **Левостороннее сопряжение и нормализация:** Каталог `products` объединяется со сжатой виртуальной таблицей через оператор `LEFT JOIN`. Это сохраняет 100% ассортиментной матрицы. Текстовые наименования унифицируются инструментом `TRIM(LOWER())` (в Python Pandas: `.str.strip().str.lower()`).
3. **Финальная зачистка витрины:** Внешний оператор `COALESCE(t2.sum, 0)` на самом верхнем уровне запроса заменяет пустые связи моста жесткими нулями (0) для нереализованных товаров. Инструмент `ORDER BY DESC` автоматически смещает позиции с нулевой активностью в хвост отчета, формируя чистую витрину лидерства продаж.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, тестирования и отладки алгоритма были зафиксированы и успешно локализованы следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Логический заусенец — внешний NULL):** Сборка сжатого подзапроса во `FROM` выполнена корректно, однако при потере финишного верхнего предохранителя `COALESCE(t2.sum, 0)` на этапе вывода отчета для нереализованных товаров левый мост выводил пустые ячейки `NULL`. 
    *   *Метод устранения:* Синтаксис верхнего уровня дополнен конструкцией `COALESCE`, принудительно преобразующей пустоту связей в цифровой ноль.
*   **Риск №2 (Сбой группировки в подзапросе):** Ошибка вызова неимеющейся колонки при попытке выполнить `GROUP BY item_name` внутри лога транзакций `sales_logs`. 
    *   *Метод устранения:* Схлопывание дубликатов переведено на строгие числовые ключи связи (`item_id`), обрабатываемые СУБД быстрее, а текстовые имена привязаны на верхнем уровне из справочника.
*   **Риск №3 (Тяжелый сбой структуры — вспучивание сумм):** Попытка прямого объединения один-ко-многим без предварительной агрегации, приведшая к раздуванию реестра дубликатами строк и искажению финансового аудита.
    *   *Метод устранения:* Конвейер перестроен на двухэтажную модель с полной изоляцией транзакционного суммирования внутри подзапроса.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование коммерческого дашборда)
Очищенная коммерческая витрина интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для мониторинга движения товарных запасов.

### Структура дашборда:
1. **Рейтинг оборачиваемости номенклатуры (ABC-анализ):** Столбчатая диаграмма (Bar Chart), ранжирующая товары по объему продаж сверху вниз, позволяющая коммерческому директору мгновенно выявлять локомотивы выручки.
2. **Реестр неликвидных позиций (Dead Stock List):** Интерактивная таблица, автоматически фильтрующая и выводящая товары с нулевым показателем продаж (`total_quantity = 0`) для оперативного принятия решений по скидкам или выводу из ассортимента.
3. **Модуль сквозного поиска номенклатуры:** Панель с фильтрацией по категориям и наименованиям товаров для быстрого аудита конкретной позиции каталога.
