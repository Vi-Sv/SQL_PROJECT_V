# Кейс №17: Оконные фильтры и изоляция лидеров категорий

## Блок: Аналитика ритейла / Коммерческий аудит
**Заказчик:** Коммерческий директор / Совет директоров.

### Бизнес-задача (ТЗ):
> *"Для оптимизации партнерской сети и проведения стратегического анализа прибыльности маркетплейса необходимо выделить ключевых контрагентов. Требуется сформировать аналитический реестр с выводом нормализованных имен агентов, наименований продуктовых направлений и чистой величины зафиксированной прибыли (пропуски занулить). По каждой отдельной товарной категории необходимо изолировать строго по одному абсолютному лидеру (Top-1), имеющему максимальный показатель доходности. Весь последующий операционный массив данных должен быть удален из итоговой выборки. Витрину упорядочить по алфавитному порядку категорий."*

---

## 🛠 Исходная структура СУБД (Таблица `agent_sales`):
*   `deal_id` (INT) — Уникальный номер коммерческой сделки (Первичный ключ).
*   `agent_name` (VARCHAR) — Наименование торгового агента / контрагента (содержит регистровый шум).
*   `category` (VARCHAR) — Продуктовая категория / направление ритейла (содержит скрытые невидимые пробелы).
*   `profit` (INT) — Величина чистой прибыли от сделки в рублях (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе технического анализа грязного ТЗ и сырого массива данных были локализованы три ключевые уязвимости:
1. **Текстовая неконсистентность признаков:** Наличие регистрового шума и скрытых пробелов в названиях категорий, способное разорвать расчет аналитических функций на некорректные группы;
2. **Пропуски финансовых метрик:** Системные пустоты `NULL` в объемах маржи блокируют работу математического движка и искажают ранжирование;
3. **Ограничение фильтрации оконных агрегатов:** Запрет спецификации ANSI SQL на прямое использование оконных функций внутри базового фильтра `WHERE` верхнего уровня.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Изоляция среды подзапроса (`FROM`):** Движок СУБД заходит во внутреннюю виртуальную матрешку таблицы `agent_sales` в оперативной памяти сервера. Комбо-резцы `TRIM(LOWER())` очищают строковые поля контрагентов и категорий, а оператор `COALESCE` трансформирует пропуски `NULL` в чистые числовые нули.
2. **Секционирование продуктовых зон:** Команда `PARTITION BY` мгновенно рассекает таблицу на изолированные, герметичные сектора по чистым названиям категорий для независимого параллельного счета.
3. **Запуск рангового автомата:** Внутри каждого сектора оператор `ORDER BY profit DESC` выстраивает сделки по убыванию финансовой маржинальности. Автоматический нумератор `ROW_NUMBER()` штампует сквозные порядковые ранги 1, 2, 3... Абсолютный лидер по прибыли получает жесткий ранг 1 (в Python Pandas логика реализована через счетчик `.cumcount() + 1`). Виртуальный куб запечатывается во внутреннем поле под техническим именем `g4` подзапроса `t2`.
4. **Хирургическое сито `WHERE` (КПП верхнего уровня):** Подзапрос закрывается скобкой и выкатывается наверх. Фильтр главного верхнего этажа `WHERE t2.g4 = 1` как хирургический скальпель срезает и выбрасывает в утиль абсолютно весь многотысячный хвост отчета, оставляя на экране строго по одному абсолютному чемпиону на каждое направление ритейла.
5. **Вывод стерильной витрины:** Финальный контур `ORDER BY 2 ASC` выстраивает итоговую готовую таблицу по алфавиту названий продуктовых категорий.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования конвейера изоляции лидеров были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за опечатки в операторе):** Падение компилятора базы на этапе парсинга из-за лишней буквы `N`, внесенной внутрь имени аналитической функции вследствие высокой штурмовой скорости ввода в блокноте (`ROW_NUNMER()`). Движок выдавал ошибку отсутствия функции в СУБД.
    *   *Метод устранения:* Опечатка моторики устранена, имя оператора приведено к промышленному каноническому стандарту `ROW_NUMBER()`.
*   **Риск №2 (Ловушка незакрытой скобки внутри оконного контура):** Сбой парсинга синтаксиса из-за пропуска парной закрывающей круглой скобки после аргумента `category` перед ключевым словом внутренней сортировки `ORDER BY`.
    *   *Метод устранения:* Контур скобок верифицирован, парный символ жестко зафиксирован в структуре окна.
*   **Риск №3 (Капкан неоднозначности поля в блоке фильтрации):** Использование голого алиаса `WHERE g4 = 1` на верхнем этаже запроса. В распределенных Highload-системах (Сбер, Яндекс) отсутствие явного родительского префикса таблицы может спровоцировать внутреннюю коллизию движка и падение вычислений.
    *   *Метод устранения:* Проведен превентивный аудит, финишное сито переведено на вызов ранга строго по полному точному адресу матрешки: `WHERE t2.g4 = 1`.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная скользящая витрина ритейла интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для презентации результатов совету директоров.

### Структура дашборда:
1. **Витрина абсолютных чемпионов направлений (Top-1 Leaderboard):** Горизонтальная столбчатая диаграмма (Horizontal Bar Chart), отображающая по одному ключевому агенту на каждую категорию, ранжированная по объему принесенной маржи (`profit`), позволяющая коммерческому директору мгновенно видеть локомотивы сети.
2. **Карта долей маржинальности товарных рынков:** Долевой график (Donut Chart), отражающий вклад прибыли каждого лидера категории в общий финансовый поток компании.
3. **Модуль интерактивного поиска контрагентов:** Фильтр-панель для моментального вывода карточки лидера и анализа структуры его сделок внутри выбранного товарного сегмента.
