# Кейс №7: Контроль дефицита цемента и сеньорская фильтрация пустых связей

## Блок: Промышленная аналитика / Логистика и Снабжение ПТО
**Заказчик:** Начальник отдела снабжения ПТО.

### Бизнес-задача (ТЗ):
> *"Для предотвращения срывов графиков строительно-монтажных работ из-за дефицита цемента и щебня необходимо сформировать экстренную ведомость материального дефицита на складах предприятия. Требуется вывести унифицированные наименования материалов из справочника и их текущие фактические объемы хранения. Критическое условие: в финальный отчет должны попасть исключительно те дефицитные позиции, которых на складе физически нет. То есть в таблице остатков по ним либо зафиксирован нулевой баланс (0), либо позиция полностью отсутствует в складских реестрах учета и сопряжение систем выдает пустой маркер NULL. Фильтрацию пустых связей реализовать строго на финишном этапе обработки после объединения данных."*

---

## Исходная структура СУБД (Сопряжение таблиц один-ко-одному/многим):

### 1. Справочник материалов (`raw_materials`):
*   `mat_code` (INT) — Уникальный код материала (Первичный ключ).
*   `mat_name` (VARCHAR) — Название строительного материала (содержит хаотичный ручной регистр и пробелы).

### 2. Остатки на складах (`warehouse_stocks`):
*   `stock_id` (INT) — Уникальный номер склада предприятия.
*   `mat_code` (INT) — Идентификатор материала (Внешний ключ сопряжения со справочником `raw_materials`).
*   `current_volume` (INT) — Текущий объем хранения в тоннах (содержит системные пропуски `NULL`).

---

## Дескриптор логики решения (Технический аудит):

В ходе предварительного технического анализа реляционных связей были локализованы две критические уязвимости и одна бизнес-коллизия:
1. **Риск аварийного падения сервера из-за несовместимости типов данных:** Попытка зашивки текстовой фильтрации и применения строкового очистителя `TRIM` к числовому полю `INT` в правых вложенных подзапросах обрушит компилятор. Числовые поля защищены от пробелов на уровне ядра СУБД.
2. **Риск фатальной потери неактивных позиций номенклатуры:** Фильтрация пустых связей внутри условия сопряжения `ON` заставит `LEFT JOIN` принудительно вывести все строки левого справочника без учета дефицита, что полностью сорвет снабженческий аудит компании.

### Алгоритм построения изящного конвейера:
1. **Развертывание моста данных:** Первичный справочник `raw_materials` объединяется с таблицей остатков `warehouse_stocks` посредством оператора `LEFT JOIN` по ключу `mat_code`. Это сохраняет 100% ассортиментной матрицы и выводит позиции, которых физически нет на балансе складов.
2. **Сеньорская пост-фильтрация пустых связей:** Фильтрация выносится строго на финальный этаж кода — в главный блок `WHERE` после сшивания моста. Логическая связка `WHERE t2.current_volume = 0 OR t2.current_volume IS NULL` намертво отсекает заполненные склады. Условие `IS NULL` изолирует внешнюю пустоту от пустых связей моста для позиций, полностью стертых из складского учета (в Python Pandas данная логика реализована через встроенный метод `.isna()`).
3. **Текстовая нормализация и зачистка витрины:** Скалярный комбо-резец `TRIM(LOWER())` на верхнем уровне `SELECT` очищает имена от пробелов и регистровых ошибок. Колонку остатков укрывает предохранитель `COALESCE`, выводящий снабженцу стерильные нули для формирования точной закупки.

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования логистического конвейера Anti-Join были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Критическое падение ядра СУБД и потеря пустых складов):** Логический сбой вследствие попытки внедрить текстовый фильтр `TRIM(current_volume) = ''` во внутренний подзапрос. Во-первых, строковые функции принудительно вызывали ошибку типов данных на поле `INT`. Во-вторых, подзапрос не захватывал отсутствующие в складском регистре позиции, возвращая ложный лог снабжения с заполненными складами инвесторам.
    *   *Метод устранения:* Фильтрация изъята из внутренних слоев и перенесена в финальный блок `WHERE` верхнего уровня.
*   **Риск №2 (Синтаксический зазор адресации функций):** Падение парсинга синтаксиса из-за ошибочного вызова префикса таблицы перед именем функции, а не перед целевым столбцом (`t1.trim(lower(mat_name))`).
    *   *Метод устранения:* Структура селекта скорректирована, адрес родительской таблицы перенесен внутрь скобок к физическому полю: `TRIM(LOWER(t1.mat_name))`.
*   **Риск №3 (Разрешение логической коллизии ТЗ):** Изначальная путаница при трактовке двусмысленного ТЗ заказчика «выведи остатки дефицита, которого нет на балансе».
    *   *Метод устранения:* Проведен сеньорский архитектурный разбор — колонка остатков выведена на витрину, но содержит строгую константу `0` для верификации факта дефицита строк и передачи реестра в отдел закупок.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная дефицитная витрина ПТО интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного контроля логистических разрывов.

### Структура дашборда:
1. **Экстренный реестр нулевых остатков (Anti-Join Table):** Интерактивная таблица с выводом полей «Чистое имя материала -> 0 тонн», отсортированная по алфавиту, служащая прямым техническим заданием на закупку для отдела снабжения.
2. **Индикатор критического дефицита (KPI Widget):** Крупный цифровой информер, отображающий общее количество номенклатурных позиций компании, находящихся в зоне абсолютного нуля (`current_volume = 0`), для экспресс-оценки рисков простоя объектов.
3. **Модуль контроля складских разрывов:** Аналитический селектор, позволяющий моментально разделять дефицит на подтвержденный (фактический ноль на складе) и системный (полное отсутствие записей о материале в базе учета).
