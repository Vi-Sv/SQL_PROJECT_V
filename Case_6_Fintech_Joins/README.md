# Кейс №6: Расчет операционной активности клиентской базы и сквозная интеграция данных

## Блок: Финтех / Бизнес-аналитика
**Заказчик:** Финансовый директор.

### Бизнес-задача (ТЗ):
> *"Необходимо сформировать сквозной консолидированный отчет по всей базе зарегистрированных клиентов. Требуется рассчитать совокупную сумму покупок за отчетный период в разрезе каждого пользователя. Важнейшее условие: клиенты, не проявившие транзакционной активности в текущем периоде, должны быть сохранены в финальном реестре без потери строк, а в поле суммы их покупок должен отображаться фактический ноль (0) вместо системных пропусков."*

---

## Исходная структура СУБД (Сопряжение двух таблиц):

### 1. Справочник клиентов (`users`):
*   `user_id` (INT) — Уникальный идентификатор пользователя (Первичный ключ).
*   `phone` (VARCHAR) — Контактный номер телефона клиента.
*   `reg_date` (TIMESTAMP) — Дата и время регистрации в системе.

### 2. Реестр транзакций (`orders`):
*   `order_id` (INT) — Уникальный номер заказа (Первичный ключ).
*   `user_id` (INT) — Идентификатор пользователя (Внешний ключ связи; содержит многократные повторения транзакций по одному клиенту).
*   `payment_amount` (INT) — Сумма чека (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Технический аудит):

В ходе проектирования архитектуры запроса были локализованы две критические уязвимости:
1. Риск «вспучивания» (кардинального размножения) строк справочника `users` и перегрузки оперативной памяти СУБД в случае прямого объединения таблиц один-ко-многим (`LEFT JOIN orders`) до этапа группировки;
2. Появление системных пустот `NULL` в итоговом отчете как следствие отсутствия связей по неактивным клиентам («молчунам») при левостороннем сопряжении.

Для построения отказоустойчивого конвейера вычислений реализован следующий алгоритм:
1. **Предварительная агрегация (Внутренний слой):** Внутри подзапроса (`or2`) к таблице логов `orders` выполняется изоляция и предварительное схлопывание дублирующих строк через `GROUP BY user_id` с одновременным расчетом суммы `SUM(COALESCE(payment_amount, 0))`. Это гарантирует, что к этапу объединения таблиц транзакционный лог сжат до уникальных ключей пользователей, что полностью исключает риск дублирования строк на верхнем уровне.
2. **Левостороннее сопряжение (Мост данных):** Исходный неприкасаемый справочник `users` сопрягается со сжатым подзапросом через оператор `LEFT JOIN` по ключу `user_id`. Это сохраняет в выборке 100% зарегистрированных клиентов, включая тех, у кого нет записей в таблице заказов.
3. **Финальная валидация метрик:** В главном `SELECT` верхнего уровня выносится предохранитель `COALESCE(or2.sum, 0)`. Это нейтрализует внешние пустоты `NULL`, образовавшиеся на стыке пустых связей моста, и выводит на витрину корректные нулевые значения для неактивных контрагентов.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование финансового дашборда)
Очищенная сквозная витрина интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для комплексного аудита клиентской базы.

### Архитектура и состав дашборда:
1. **Распределение клиентской активности (KPI Cards):** Блок из двух цифровых виджетов: «Общий объем выручки компании» и «Количество абсолютно неактивных клиентов за период» для мгновенной оценки удержания (Retention).
2. **Диаграмма концентрации продаж (Доли рынка):** График, отображающий долю выручки от топ-10 активных пользователей в общем объеме коммерческого потока.
3. **Сквозной интерактивный реестр (Data Table):** Сводная таблица с полями «ID -> Телефон -> Сумма» со встроенным поиском и фильтром по нулевым значениям для оперативной передачи списка неактивных клиентов в отдел маркетинга.
