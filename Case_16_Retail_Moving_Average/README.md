# Кейс №16: Математическое сглаживание трендов и расчет скользящего среднего

## Блок: Аналитика ритейла / Управление продажами
**Заказчик:** Коммерческий директор / Совет инвесторов.

### Бизнес-задача (ТЗ):
> *"Для нивелирования краткосрочных хаотичных колебаний выручки и выявления реальных коммерческих трендов необходимо реализовать алгоритм математического сглаживания. Требуется сформировать реестр торговых дней по филиалам сети с выводом номеров магазинов, дат операций и чистой выручки (пропуски занулить). В смежной колонке необходимо спроектировать расчет скользящего среднего значения выручки за трехдневный календарный период (текущие сутки и 2 предыдущих дня). Расчет должен опираться строго на временные интервалы, а не физические строки, для исключения искажений в случае закрытия торговых точек на ремонт или отсутствия транзакций. Вывод упорядочить по магазинам и хронологии."*

---

## Исходная структура СУБД (Таблица `daily_shop_revenue`):
*   `row_id` (INT) — Уникальный номер строки лога (Первичный ключ).
*   `shop_id` (INT) — Идентификатор торговой точки / магазина сети.
*   `report_date` (TIMESTAMP) — Календарная дата и время формирования отчета.
*   `daily_revenue` (INT) — Объем суточной выручки филиала в рублях (содержит пропуски `NULL`).

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе технического анализа сырого массива данных были локализованы две системные уязвимости:
1. **Риск декомпозиции финансовой метрики:** Наличие пропусков `NULL` в ячейках оборота блокирует работу агрегаторов;
2. **Логический капкан физических строк (`ROWS`):** Если в логах присутствуют разрывы дат (например, точка не работала в праздники), стандартная конструкция `ROWS BETWEEN 2 PRECEDING` принудительно подтянет физические строки недельной давности, полностью разрушив финансовую точность трехдневного тренда для инвесторов.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Изоляция среды (`FROM`):** Движок СУБД разворачивает сырой транзитный массив таблицы `daily_shop_revenue` в оперативной памяти сервера для последующей обработки.
2. **Обеспечение качества данных:** Оператор `COALESCE(daily_revenue, 0)` на входе в конвейер выжигает черные дыры `NULL` нулями, подготавливая стерильную базу для вычислений.
3. **Секционирование торговых контуров:** Конструкция `PARTITION BY shop_id` нарезает таблицу на герметичные, изолированные сектора по номерам торговых точек. Внутри каждого магазина даты выстраиваются хронологическим вектором вперед с помощью `ORDER BY report_date`.
4. **Развертывание ползающего лифта времени (`RANGE`):** Движок СУБД активирует логический временной резец `RANGE BETWEEN INTERVAL '2' DAY PRECEDING AND CURRENT ROW`. Для каждой текущей строки сервер считывает метку времени `TIMESTAMP`, отсчитывает ровно 2 календарных дня назад (72 часа) и вычисляет средний чистый оборот `AVG` исключительно за этот календарный шаг, сглаживая тренд и игнорируя физические разрывы строк (в Python Pandas логика зеркально реализована через группировку и специализированный временной метод `.rolling('3D')`). Результат фиксируется под именем `rolling_average_3d`.
5. **Вывод витрины:** Финальный отчет выстраивается по порядку магазинов и строгому возрастанию дат.

## 🛠 ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования скользящего конвейера были зафиксированы и устранены следующие архитектурные риски и ошибки сбоя:

*   **Риск №1 (Косметический зазор моторики — пропуск имени витрины):** Сбой парсинга и падение компилятора из-за упущенного выходного имени алиаса после закрытия скобки оконной конструкции `OVER ()`. Ключевое слово `AS` было прописано, но имя столбца стерто вследствие высокой скорости кодинга в конце сессии.
    *   *Метод устранения:* Структура селекта отрегулирована, целевое имя `rolling_average_3d` интегрировано на верхний этаж вывода.
*   **Риск №2 (Тяжелый логический сбой структуры — капкан физических строк):** Первоначальная попытка проектирования скользящего среднего через рамки физических строк `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW`. Тестирование на грязных данных выявило фатальное искажение трендов: при наличии нескольких транзакций в один день или при долгосрочных простоях магазина функция подтягивала данные некорректных периодов.
    *   *Метод устранения:* Конвейер переведен на строгую временную рамку `RANGE BETWEEN INTERVAL '2' DAY PRECEDING`, стабилизировавшую финансовую точность отчетов.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование коммерческого дашборда)
Очищенная скользящая витрина ритейла интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для стратегического сглаживания трендов продаж.

### Структура дашборда:
1. **График сглаженного тренда выручки (Smooth Line Chart):** Комбинированная диаграмма, где тонкими полупрозрачными линиями выведены хаотичные ежедневные скачки выручки (`clean_revenue`), а живой контрастной линией наложено трехдневное скользящее среднее значение. Позволяет совету инвесторов видеть реальное направление бизнеса.
2. **Индикатор межрегиональной стабильности ФОТ/Выручки:** Сводный информер, оценивающий стандартное отклонение скользящей средней по торговым точкам холдинга для выявления затухающих локаций.
3. **Панель переключения временных интервалов:** Селектор дат для быстрого масштабирования скользящего графика от недели до года.
