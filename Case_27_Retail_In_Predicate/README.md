# Кейс №27: Множественные подзапросы и сито предикатов IN

## Блок: Аналитика ритейла / Коммерческий маркетинг
**Заказчик:** Директор по маркетингу / Коммерческий директор холдинга.

### Бизнес-задача (ТЗ):
> *"Для анализа структуры коммерческой эффективности южной торговой зоны и выявления скрытых закономерностей продаж необходимо сформировать аналитическую витрину доходов. Требуется провести предварительный отбор городов, которые географически относятся строго к макрорегиону 'юг' и зафиксировали ежемесячную прибыль свыше 250 000 единиц. На финальный экран необходимо вывести полный лог всех торговых показателей по данным городам-лидерам без нагромождения условий и усложнения структуры тяжелыми склейками таблиц. Все наименования нормализовать, вывод отсортировать по убыванию объемов прибыли."*

---

## Исходная структура СУБД (Таблица `branch_revenues`):
*   `branch_id` (INT) — Уникальный номер торгового филиала (Первичный ключ).
*   `city_name` (VARCHAR) — Наименование города (содержит грязный ручной ввод регистров и пробелов).
*   `monthly_profit` (INT) — Величина ежемесячной прибыли в рублях (содержит пропуски `NULL`).
*   `region_zone` (VARCHAR) — Географическая макрозона / регион присутствия.

---

## Дескриптор логики решения (Анатомия движка СУБД):

В ходе проектирования архитектуры аналитического скрипта была применена модель предикативной фильтрации множеств на основе обобщенного табличного выражения (CTE) и оператора `IN`. Данный паттерн позволяет изолировать логику отбора на нижнем уровне и избежать тяжелых сопряжений таблиц (`JOIN`) один-ко-многим на верхнем этаже запроса, что полностью исключает риск раздувания строк и снижает нагрузку на дисковую подсистему базы данных.

### Пошаговый алгоритм выполнения запроса движком СУБД:
1. **Инициализация базового цеха (`WITH top_profit_subquery`):** Сервер СУБД разворачивает изолированное виртуальное пространство в оперативной памяти (ОЗУ). Комбо-резцы `TRIM(LOWER())` нормализуют строковые поля городов и регионов, а `COALESCE` предохраняет математику от пропусков. 
2. **Сжатие массива множества:** Условие `DISTINCT` на входе в CTE намертво сжимает выборку до уникальных названий целевых южных городов-лидеров, прошедших финансовый фильтр, избавляя оперативную память от дубликатов строк.
3. **Сканирование физического источника:** Главный внешний запрос начинает параллельное считывание исходной таблицы `branch_revenues`.
4. **Запуск сита предикатов (`WHERE IN`):** Оператор `IN` работает как динамический внутренний фильтр верхнего уровня (в Python Pandas логика реализована через векторный метод `.isin()`). Он мгновенно сверяет текущий зачищенный город строки со списком уникальных локаций-лидеров, полученных из CTE. Строки, не прошедшие верификацию множества, отсекаются.
5. **Вывод стерильной витрины:** Итоговый финансовый реестр выстраивается ровными рельсами по строгому убыванию прибыли посредством конструкции `ORDER BY 2 DESC` (сортировка по порядковому номеру столбца).

## ЖУРНАЛ ОТЛАДКИ И КАНАЛИЗАЦИЯ ОШИБОК (Debug Log)
В процессе проектирования, сборки и тестирования предикативного конвейера ритейла были зафиксированы и устранены следующие технические риски и ошибки сбоя:

*   **Риск №1 (Синтаксическая отсечка из-за капкана лишней запятой):** Падение компилятора базы данных на этапе парсинга из-за разделительного символа запятой (`,`), ошибочно влепленного после объявления последнего поля внутри подзапроса непосредственно перед оператором `FROM` (`AS g1, FROM`). 
    *   *Метод устранения:* Зазор моторики устранен, избыточный разделительный знак удален из структуры селекта.
*   **Риск №2 (Ловушка потерянной закрывающей скобки главного цеха CTE):** Критическая синтаксическая ошибка на стыке слоев, при которой внешний главный запрос ложно слился с телом подзапроса из-за пропуска круглой скобки, завершающей контур СТЕ-магистрали `top_profit_subquery`. Скобка условия `WHERE` была закрыта, но внешняя цепь осталась разомкнутой.
    *   *Метод устранения:* Структура скобочного каркаса верифицирована Ключом №3, парный закрывающий символ интегрирован перед верхним `SELECT`.
*   **Риск №3 (Защита орфографии текстового дескриптора):** Фиксация механических помарок и опечаток в текстовом описании архитектуры («марешке» вместо «матрешке», «вреенного» вместо «временного»), вызванных скрытным скоростным вводом с телефона в условиях административного надзора. На уровне кода SQL-чистовика технические риски отсутствовали.

## СЛОЙ ВИЗУАЛИЗАЦИИ И BI (Проектирование аналитического дашборда)
Очищенная предикативная витрина ритейла из каскада подзапросов интегрируется с BI-платформой (Apache Superset / Yandex DataLens) для оперативного контроля эффективности регионов.

### Структура дашборда:
1. **Географический реестр городов-лидеров (Target Profit Table):** Детализированная таблица «Зачищенный город -> Прибыль», автоматически аккумулирующая филиалы, прошедшие сквозное сито соответствия, отсортированная по убыванию, для визуального контроля маржинальных зон.
2. **Карта распределения доходов южной зоны:** Линейно-столбчатая диаграмма (Bar Chart), сопоставляющая объемы ежемесячной прибыли отфильтрованных городов для выявления локаций-локомотивов.
3. **Модуль быстрой номенклатурной селекции:** Фильтр-панель для моментального вывода карточек конкретных торговых точек внутри выделенного города-лидера.
